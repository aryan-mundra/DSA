Q1 — Torch Puzzle — Bridge Crossing

LOGIC / REASONING

4 people must cross a bridge at night with one torch. The bridge holds at most 2 people at a time. When two cross together, they travel at the slower person's speed. The torch must be carried back each time.

Speeds: A = 1 min, B = 2 min, C = 5 min, D = 10 min.
Find: The minimum total time to get everyone across.

Key Insight

The naive strategy (always send the fastest person back) gives 19 minutes. The optimal trick is to pair the two slowest people together so their large crossing times overlap instead of adding up separately. Use the two fastest as "escorts".

Optimal Steps

StepActionTimeLeft SideRight Side
1 A & B cross → 2 min C, D A, B
2 ← A comes back 1 min A, C, D B
3 C & D cross → 10 min A B, C, D
4 ← B comes back 2 min A, B C, D
5 A & B cross → 2 min — A, B, C, D
Total 2 + 1 + 10 + 2 + 2 = 17 minutes

Why not the naive approach?

If A escorts everyone individually: A&B (2) + A back (1) + A&C (5) + A back (1) + A&D (10) = 19 minutes.
The optimal solution saves 2 minutes by making C and D cross together (paying 10 once instead of 5 + 10 = 15).

Answer: The minimum time is 17 minutes.

Q2 — Best Time to Buy and Sell Stock

JAVA

Given an array prices where prices[i] is the stock price on day i, find the maximum profit from one buy-sell transaction. If no profit is possible, return 0.
Examples: [7,1,5,3,6,4] → 5  |  [7,6,4,3,1] → 0

Intuition

The key idea: We want to buy low and sell high. As we go through each day's price, we need to remember the cheapest price we've seen so far (that's our best buying opportunity).

For every price we see, we ask: "If I had bought at the cheapest price so far, and I sell today, what profit would I make?" We keep track of the best profit we find.

How It Works

  1. Start with minPrice = infinity (very high number) and maxProfit = 0
  2. Go through each day's price:
    • If today's price is lower than our minPrice, update it (we found a better buying day)
    • Otherwise, calculate profit = today's price - cheapest price we've seen
    • If this profit is better than our current best, remember it
  3. Return the best profit we found

Example: [7, 1, 5, 3, 6, 4]

  • Day 0: price=7 → minPrice=7, profit=0
  • Day 1: price=1 → minPrice=1 (cheaper!), profit=0
  • Day 2: price=5 → profit = 5-1 = 4 (new best!)
  • Day 3: price=3 → profit = 3-1 = 2 (not better than 4)
  • Day 4: price=6 → profit = 6-1 = 5 (new best!)
  • Day 5: price=4 → profit = 4-1 = 3 (not better than 5)
  • Answer: 5
public int maxProfit(int[] prices) {
    int minPrice = Integer.MAX_VALUE;
    int maxProfit = 0;

    for (int price : prices) {
        if (price < minPrice) {
            minPrice = price;
        } else {
            maxProfit = Math.max(maxProfit, price - minPrice);
        }
    }
    return maxProfit;
}
Time: O(n) — single pass  |  Space: O(1)

Q3 — Rank Transform of an Array

JAVA

Replace each element with its rank. Rank starts at 1, larger elements get larger ranks. Equal elements share the same rank. Ranks should be as small as possible.
Examples: [100,50,70,90] → [4,1,2,3]  |  [10,10,10] → [1,1,1]

Intuition

The problem: We need to give each number a rank (1, 2, 3, ...) where smaller numbers get smaller ranks. If two numbers are the same, they should get the same rank.

The trick: If we sort the array first, the smallest value will be first (rank 1), the next unique value will be rank 2, and so on. We can build a "lookup table" (HashMap) that maps each unique value to its rank.

How It Works (Step-by-Step)

  1. Make a copy and sort it: [100, 50, 70, 90] → sorted: [50, 70, 90, 100]
  2. Build a rank map: Go through the sorted array:
    • 50 is first unique value → rank 1
    • 70 is next → rank 2
    • 90 → rank 3
    • 100 → rank 4
    • Store in HashMap: {50→1, 70→2, 90→3, 100→4}
  3. Map original values to ranks: Go through the original array [100, 50, 70, 90]:
    • 100 → look up in map → rank 4
    • 50 → rank 1
    • 70 → rank 2
    • 90 → rank 3
    • Result: [4, 1, 2, 3]

Why use a HashMap? Once we build the rank map, we can look up any value's rank in O(1) time (instant lookup).

Handling duplicates: If we see [10, 10, 10], after sorting we see 10 three times. We only assign rank 1 to the first 10, and skip the duplicates. So all three 10s get rank 1.

import java.util.*;

public int[] transformRank(int[] arr) {
    int[] sorted = arr.clone();
    Arrays.sort(sorted);

    Map<Integer, Integer> rankMap = new HashMap<>();
    int rank = 1;

    for (int val : sorted) {
        if (!rankMap.containsKey(val)) {
            rankMap.put(val, rank++);
        }
    }

    int[] result = new int[arr.length];
    for (int i = 0; i < arr.length; i++) {
        result[i] = rankMap.get(arr[i]);
    }
    return result;
}
Time: O(n log n) due to sorting  |  Space: O(n)

Q4 — Efficient Time-Range Queries for Sensor Data

JAVA

Store millions of sensor readings (sensorId, timestamp, value) in memory. Given a sensorId and a time range [startTime, endTime], efficiently retrieve all matching readings.

Intuition

The challenge: We have millions of sensor readings. Each reading has a sensor ID, a timestamp, and a value. We need to quickly find all readings from a specific sensor within a time range (e.g., "Give me all readings from sensor S1 between 10:00 and 11:00").

Why not just use a list? If we store all readings in a simple list and search linearly, it would take O(n) time (checking every reading) — way too slow for millions of readings!

The Solution (Nested Data Structure)

We use a two-level structure to organize data efficiently:

  1. Level 1 — HashMap by sensor ID:
    • Key: sensor ID (e.g., "S1", "S2")
    • Value: a TreeMap containing all readings for that sensor
    • Why? HashMap gives us O(1) instant lookup to find a specific sensor's data
  2. Level 2 — TreeMap by timestamp:
    • Key: timestamp (e.g., 10:15, 11:30)
    • Value: a List of all readings at that exact time
    • Why? TreeMap keeps timestamps sorted, so we can efficiently get a range of times using subMap()

How Searching Works

  1. Step 1: Use HashMap to find the sensor's TreeMap → O(1)
  2. Step 2: Use TreeMap's subMap(startTime, endTime) to get all timestamps in the range → O(log n) to find boundaries
  3. Step 3: Iterate through the matching timestamps and collect all readings → O(k) where k = number of results
  4. Total: O(log n + k) — much faster than O(n)!

Visual Example

store = {
  "S1" → TreeMap {
    10:00 → [Reading1, Reading2],
    10:15 → [Reading3],
    11:30 → [Reading4]
  },
  "S2" → TreeMap {
    10:03 → [Reading5]
  }
}

Query: search("S1", 10:00, 11:00)
→ Get S1's TreeMap (O(1))
→ subMap(10:00, 11:00) returns {10:00→[...], 10:15→[...]}
→ Collect all readings from those entries
→ Return [Reading1, Reading2, Reading3]

Why this is fast: We never scan through irrelevant sensors or timestamps. We jump directly to the right sensor, then jump to the right time range using the sorted TreeMap.

import java.time.LocalTime;
import java.util.*;

class SensorReading {
    String sensorId;
    LocalTime timestamp;
    double value;

    public SensorReading(String sensorId, LocalTime timestamp, double value) {
        this.sensorId = sensorId;
        this.timestamp = timestamp;
        this.value = value;
    }
}

class SensorStore {
    private Map<String, TreeMap<LocalTime, List<SensorReading>>> store = new HashMap<>();

    public void addReading(SensorReading reading) {
        /* LIVE EXAMPLE: Suppose we're adding Reading(sensorId="S1", timestamp=10:00, value=25.5)
         * 
         * GOAL: Store it in this structure:
         * store = {
         *   "S1" → TreeMap {
         *     10:00 → List [Reading1, Reading2, ...]
         *   }
         * }
         */

        // STEP 1: Check if sensor "S1" already has a TreeMap in our store
        TreeMap<LocalTime, List<SensorReading>> sensorData = store.get(reading.sensorId);
        
        // STEP 2: If sensor doesn't exist yet (first time seeing "S1")
        if (sensorData == null) {
            // Create an empty TreeMap for this sensor
            sensorData = new TreeMap<>();
            
            // Add it to main store: store.put("S1", new TreeMap)
            // Now store = { "S1" → TreeMap{} }
            store.put(reading.sensorId, sensorData);
        }
        // After this point, sensorData definitely exists and points to S1's TreeMap
        
        // STEP 3: Now check if timestamp 10:00 exists in S1's TreeMap
        List<SensorReading> readingsAtTime = sensorData.get(reading.timestamp);
        
        // STEP 4: If this timestamp doesn't exist yet (first reading at 10:00 for S1)
        if (readingsAtTime == null) {
            // Create an empty list for this timestamp
            readingsAtTime = new ArrayList<>();
            
            // Add it to S1's TreeMap: sensorData.put(10:00, new ArrayList)
            // Now store = { "S1" → TreeMap{ 10:00 → [] } }
            sensorData.put(reading.timestamp, readingsAtTime);
        }
        // After this point, readingsAtTime definitely exists and points to the list at 10:00
        
        // STEP 5: Add our reading to the list
        // readingsAtTime.add(Reading(S1, 10:00, 25.5))
        // Final: store = { "S1" → TreeMap{ 10:00 → [Reading(S1,10:00,25.5)] } }
        readingsAtTime.add(reading);
        
        /* WHAT IF we add another reading at same time?
         * Example: Reading(sensorId="S1", timestamp=10:00, value=30.2)
         * 
         * - Step 1: sensorData = store.get("S1") → Found! (TreeMap already exists)
         * - Step 2: Skip (not null)
         * - Step 3: readingsAtTime = sensorData.get(10:00) → Found! (List already exists)
         * - Step 4: Skip (not null)
         * - Step 5: Add to existing list
         * 
         * Result: store = { "S1" → TreeMap{ 10:00 → [Reading1, Reading2] } }
         */
    }

    public Collection<SensorReading> search(String sensorId,
                                              LocalTime startTime,
                                              LocalTime endTime) {
        List<SensorReading> result = new ArrayList<>();

        TreeMap<LocalTime, List<SensorReading>> sensorData = store.get(sensorId);
        if (sensorData == null) return result;

        // subMap returns entries in [startTime, endTime] inclusive
        NavigableMap<LocalTime, List<SensorReading>> subMap =
            sensorData.subMap(startTime, true, endTime, true);

        for (List<SensorReading> readings : subMap.values()) {
            result.addAll(readings);
        }
        return result;
    }
}
Time Complexity: O(log n + k) — where n = number of distinct timestamps for that sensor, k = number of readings in the range.
Space: O(total readings across all sensors).

Understanding NavigableMap and Its Methods

What is NavigableMap? It's an interface (blueprint) that TreeMap implements. It provides special "navigation" methods to move around a sorted map efficiently.

Key Methods You Should Know:

  • subMap(fromKey, fromInc, toKey, toInc): Get all entries in a range
    • Example: map.subMap(10:00, true, 12:00, false) → entries from 10:00 (inclusive) to 12:00 (exclusive)
  • lowerKey(key): Find the largest key less than the given key
    • Example: If map has {10:00, 10:15, 10:30}, lowerKey(10:15) → 10:00
  • higherKey(key): Find the smallest key greater than the given key
    • Example: higherKey(10:15) → 10:30
  • firstEntry(): Get the entry with the smallest key
    • Example: firstEntry() → {10:00=[readings...]}
  • lastEntry(): Get the entry with the largest key
    • Example: lastEntry() → {10:30=[readings...]}
  • floorKey(key): Largest key ≤ given key
  • ceilingKey(key): Smallest key ≥ given key

💡 Did You Know?

Understanding the true/false parameters in subMap():

The boolean values control whether boundaries are included or excluded:

  • subMap(9:00, true, 11:00, true) → [9:00 to 11:00] — both boundaries included
  • subMap(9:00, false, 11:00, true) → (9:00 to 11:00] — exclude 9:00, include 11:00
  • subMap(9:00, true, 11:00, false) → [9:00 to 11:00) — include 9:00, exclude 11:00
  • subMap(9:00, false, 11:00, false) → (9:00 to 11:00) — both excluded

Quick Reference Code:

TreeMap<LocalTime, String> times = new TreeMap<>();
times.put(LocalTime.of(9, 0), "9am");
times.put(LocalTime.of(10, 15), "10:15am");
times.put(LocalTime.of(11, 30), "11:30am");

// Get times from 10:00 to 11:30 (inclusive)
NavigableMap<LocalTime, String> range = 
    times.subMap(LocalTime.of(10, 0), true, LocalTime.of(11, 30), true);
// Result: {10:15="10:15am", 11:30="11:30am"}

// Find what comes before 11:00
LocalTime prev = times.lowerKey(LocalTime.of(11, 0));  // → 10:15

// Find what comes after 10:15
LocalTime next = times.higherKey(LocalTime.of(10, 15)); // → 11:30

Why use NavigableMap type? The subMap() method returns a view (not a TreeMap object), so we use the interface type. This follows Java best practices.

Q5 — Student Club Management — Database Schema & SQL

SQL

Design a 3NF database for a university's student club system. Students can join multiple clubs, each club has a president and organizes events. Then answer three SQL queries.

Understanding the Problem

What we're building: A database to manage student clubs at a university. We need to track:

  • Which students belong to which clubs (many-to-many: one student can join multiple clubs, one club has multiple students)
  • Who is the president of each club
  • Which events each club organizes
  • Which students attended which events (many-to-many: one student can attend multiple events, one event has multiple attendees)

Why 5 Tables? (Breaking Down the Design)

  1. students — Core table storing basic student info (ID, name, enrollment year)
  2. clubs — Core table storing club info (name, who's president)
    • President is a foreign key to students table (president must be a valid student)
  3. student_clubs — Junction table connecting students ↔ clubs
    • Why needed? A student can join many clubs, and a club has many members
    • Each row represents one membership: (student_id, club_id)
  4. events — Stores each event organized by a club
    • Each event belongs to exactly one club (foreign key to clubs)
  5. event_participants — Junction table connecting students ↔ events
    • Why needed? A student can attend many events, and an event can have many attendees
    • Each row represents one attendance: (event_id, student_id)

What is 3NF? (Simple Explanation)

Third Normal Form (3NF) means our database has no redundancy and data is organized properly:

  • Rule 1: Every column depends on the primary key (no unrelated data in a table)
  • Rule 2: No transitive dependencies (A → B → C, where B is not a key)
  • Example of violation: If we stored "club_name" in the events table, we'd repeat the same club name for every event. Instead, we store only club_id and look up the name when needed.

Our design is in 3NF because each table stores only directly related information, and we use foreign keys to connect related data without duplication.

CREATE TABLE students (
    student_id      INT          PRIMARY KEY,
    name            VARCHAR(100) NOT NULL,
    enrollment_year INT          NOT NULL
);

CREATE TABLE clubs (
    club_id      INT          PRIMARY KEY AUTO_INCREMENT,
    club_name    VARCHAR(100) NOT NULL UNIQUE,
    president_id INT          NOT NULL,
    FOREIGN KEY (president_id) REFERENCES students(student_id)
);

CREATE TABLE student_clubs (
    student_id INT NOT NULL,
    club_id    INT NOT NULL,
    PRIMARY KEY (student_id, club_id),
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (club_id)    REFERENCES clubs(club_id)
);

CREATE TABLE events (
    event_id   INT          PRIMARY KEY AUTO_INCREMENT,
    event_name VARCHAR(200) NOT NULL,
    event_date DATE         NOT NULL,
    club_id    INT          NOT NULL,
    FOREIGN KEY (club_id) REFERENCES clubs(club_id)
);

CREATE TABLE event_participants (
    event_id   INT NOT NULL,
    student_id INT NOT NULL,
    PRIMARY KEY (event_id, student_id),
    FOREIGN KEY (event_id)   REFERENCES events(event_id),
    FOREIGN KEY (student_id) REFERENCES students(student_id)
);

Query 1 — List all students with the clubs they belong to (include students with no club)

SELECT s.student_id,
       s.name        AS student_name,
       c.club_name
FROM   students s
LEFT JOIN student_clubs sc ON s.student_id = sc.student_id
LEFT JOIN clubs c          ON sc.club_id = c.club_id
ORDER BY s.name, c.club_name;

Query 2 — Club president name and number of events organized by their club in 2024

SELECT s.name          AS president_name,
       c.club_name,
       COUNT(e.event_id) AS events_in_2024
FROM   clubs c
JOIN   students s ON c.president_id = s.student_id
LEFT JOIN events e ON c.club_id = e.club_id
                   AND YEAR(e.event_date) = 2024
GROUP BY c.club_id, c.club_name, s.name
ORDER BY events_in_2024 DESC;

Query 3 — All event details with participant names (include events with no attendees)

SELECT e.event_name,
       e.event_date,
       c.club_name,
       s.name AS participant_name
FROM   events e
JOIN   clubs c              ON e.club_id = c.club_id
LEFT JOIN event_participants ep ON e.event_id = ep.event_id
LEFT JOIN students s           ON ep.student_id = s.student_id
ORDER BY e.event_date, e.event_name, s.name;
