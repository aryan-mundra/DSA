1. Why are datatypes important in Java?

Why are datatypes important in Java?

Java is strongly typed: you must say what kind of data each variable holds. That matters because (1) different types use different memory — e.g. int is smaller than long; (2) the compiler catches mistakes early (wrong assignments); (3) each type allows only sensible operations (math on numbers, string methods on text); (4) names like int age = 25 make code easy to read; (5) you pick the right shape for your data (whole number, decimal, text, true/false).

Real life: Like sorting laundry into labelled baskets — socks vs shirts — so nothing is washed the wrong way.

Indirect (follow-up) questions

What would happen if you try to add a String and an int in Java? Why?

Java turns the int into text and joins: "Age" + 25 gives "Age25". It is string concatenation, not math.

Is it good to use String isActive = "true" instead of boolean isActive = true?

Prefer boolean. A String can hold any text, so typos and invalid values slip in. boolean is only true or false.

Is Java a strongly typed language?

Yes. Every variable has a declared type before use.

2. What is reference type in Java?

What is reference type in Java?

Reference types (non-primitives) store a reference (address) to an object in memory, not the whole object inside the variable. They can be null, have methods, and objects usually live on the heap.

Real life: Your contact entry stores a phone number that points to a person — the entry is not the person.

Indirect (follow-up) questions

null.

Is String a primitive data type or an object (reference type) in Java?

It is an object (reference type).

What happens when you assign one object reference variable to another object reference variable in Java?

Both variables point to the same object in memory.

In how many ways can we create the object?

Common ways: (1) new — normal; constructor runs. (2) clone() — copy; constructor usually does not run. (3) Deserialization (ObjectInputStream.readObject()) — constructor does not run. (4) Reflection (Constructor.newInstance()) — constructor does run.

Which interface must a class implement in order to clone objects using Java’s cloning mechanism?

4. Explain the four pillars of OOP (OOPS)

Explain the four pillars of object-oriented programming (OOPS).

Inheritance: A child class reuses fields and methods from a parent. Java allows one parent class but many interfaces.

Encapsulation: Bundle data and methods; hide fields with private and expose safe access (getters/setters).

Polymorphism: “Many forms” — same idea, different behaviour. Overloading = compile-time; overriding = runtime.

Abstraction: Hide inner details; show only what users need — using abstract classes and interfaces.

Real life: Inheritance: “a car is a vehicle.” Polymorphism: one “horn” button, different sounds per vehicle.

5. How is abstraction different from encapsulation?

How is abstraction different from encapsulation?

Abstraction hides how work is done and shows simple controls. Example: a washing machine — you press start; you do not wire the motor yourself. In code: interfaces / abstract types hide implementation.

Encapsulation hides data and protects it from random changes. Example: a car hides engine sensors; the driver uses pedals only. In code: private fields + controlled methods.

6. Questions on encapsulation

How do we achieve encapsulation in Java?

Keep fields private and expose public getters/setters (or other public methods) so outside code cannot touch data directly.

Why is encapsulation needed?

It protects data from bad or accidental changes, keeps data and the rules together, and makes programs safer and easier to change.

Why do we make global variables (or fields) private in Java?

If fields are public, any code can change them anytime and break rules. Private fields force everyone through your methods.

What is the role of setters and getters in encapsulation?

Getters let others read values safely. Setters let them change values only the way you allow (checks, limits).

If no encapsulation is implemented, what consequences may occur?

Any part of the app can change data; bugs spread and are hard to trace. One small change can break distant code.

Can you provide an example that illustrates the concept of encapsulation?

Bank account: balance is private. People use deposit() and withdraw() only. You can block negative balance inside those methods — nobody edits balance directly.

How does encapsulation contribute to code organization and maintenance?

Each class owns its data and rules. You can change one class without breaking the whole app. New readers understand one box at a time.

In what ways does encapsulation enhance code security?

Sensitive data is not exposed. Only allowed actions run through your methods — like a locked drawer and one official key.

What are the potential drawbacks of overusing encapsulation in a program?

Too many tiny getters/setters add noise. Use encapsulation where it really protects or clarifies data — not everywhere by habit.

7. Questions on inheritance

Why do we need inheritance in object-oriented programming?

To reuse code and build clear hierarchies instead of copying the same logic into many classes.

How do we make inheritance happen in Java code?

Use extends, e.g. class Dog extends Animal. The child gets public and protected members from the parent.

What does the extends keyword really mean in Java?

“I reuse the public and protected parts of that parent class.”

Can you give a real-life example of inheritance?

Vehicle has “move” and “carry people.” Car extends Vehicle and adds AC and music — shared behaviour stays in one place.

Why can't we use multiple inheritance for classes in Java?

The “diamond problem”: two parents with the same method confuse which code to run. Java avoids that for classes.

Can we do inheritance in Java without using the extends keyword?

Yes — implement interfaces, or use composition (hold another object inside your class).

Are constructors and private members inherited?

Constructors are not inherited; the child defines its own and may call super(...). Private members are not visible in the child; access them through parent public/protected methods if needed.

How do you stop a class from being inherited?

Declare the class final, or mark methods final to stop overriding.

Why does Java allow multiple interfaces but not multiple classes?

Interfaces are contracts — the class supplies one implementation. Multiple parent classes could each bring real method bodies that clash; one parent keeps that simple.

8. What is constructor in Java?

What is constructor in Java?

A special block with the same name as the class and no return type (not even void). It runs when you create an object (usually with new) to set starting values. You can overload constructors.

Real life: Filling a new employee form on day one — name, ID, desk — before they start work.

Indirect (follow-up) questions

What is the difference between a constructor and a normal method in Java?

Constructor runs at object creation, has no return type, name = class name. Methods can run anytime, have any name, and may return values.

Can a class have more than one constructor in Java?

Yes — constructor overloading with different parameter lists.

What is the difference between a default constructor and a parameterized constructor?

Default: no parameters, sets defaults (or compiler adds one if you wrote none). Parameterized: takes arguments to set specific initial state.

Can a constructor in Java have a return type? Is void allowed as the return type of a constructor?

No. If you add a return type, Java treats it as a normal method, not a constructor.

How is a constructor invoked in Java?

Automatically when you create an object, e.g. new Student("John").

What happens if you don't provide a constructor in a class?

If you define no constructors, Java adds a default no-arg constructor. If you define any constructor yourself, the compiler does not add the default unless you write it.

Can you have both a default and a parameterized constructor in the same class?

Yes — very common. The call matches the argument list.

What is constructor overloading?

Multiple constructors in one class with different parameters so objects can be built in different ways.

Can a constructor call another constructor of the same class? How?

Yes — this(...) as the first line (constructor chaining).

What is a static block, and how is it related to constructors?

A static block runs once when the class loads, before any object exists. It initializes static data. It is not the same as a constructor (constructors run per object).

What is the difference between an instance initialization block and a constructor?

Instance init block runs before the constructor on every object creation; good for shared setup. Constructors are still the main flexible place to initialize.

Can we override a constructor?

No — constructors are not inherited.

Can we overload a constructor?

Yes.

Can a constructor be static or final?

No to both — constructors belong to object creation, not the class slot, and are not inherited so final does not apply like for methods.

9. Difference between == and .equals() in Java

What is the difference between == and .equals() in Java?

== compares references for objects: “same object in memory?” It does not compare inner meaning by itself.

equals() compares logical content when the class overrides it (e.g. String compares character sequence).

Two new String("hi") may give false for == but true for equals().

Literal (string pool), new String("..."), StringBuilder/StringBuffer then toString(), new String(char[]), and related APIs.

What is the String constant pool?

A special area where string literals are reused so identical text can share one object — saves memory because strings are immutable.

What does the String.intern() method do, and why is it useful?

It returns the pooled copy (and adds the string to the pool if needed). Useful to dedupe heap strings and save memory when you care about identity in the pool.

12. Is Java pass-by-value or pass-by-reference?

Is Java pass-by-value or pass-by-reference?

Java is always pass-by-value. For primitives, the value is copied — changes inside a method do not affect the caller’s variable. For objects, the reference is copied — you can still change the object’s fields through that copy, but reassigning the parameter does not change the caller’s variable.

Real life: Primitives: you hand a photocopy of a number. Objects: you hand a photocopy of an address — both papers still lead to the same house.

13. What is immutability, and how to achieve it?

What is immutability, and how to achieve it?

An immutable object’s state does not change after creation; “updates” create new objects (e.g. String). To design an immutable class: make the class final, fields private final, no setters, and do not return mutable internals directly — return copies (e.g. new list from getter).

Indirect (follow-up) questions

How is immutability different from the final keyword?

final on a variable means you cannot reassign the variable. Immutability means the object’s internal state cannot change.

If a reference variable is final, does that mean the object it points to is immutable?

Not always. final only stops reassignment of the reference; the object might still be mutable inside.

If a class has a mutable field (for example an ArrayList), how do you stop callers from modifying that list when they use a getter?

Do not return the internal list directly from a getter; return new ArrayList<>(list) or an unmodifiable view.

How do you make a class thread-safe without using the synchronized keyword?

Prefer immutability — no shared changing state, so threads cannot corrupt it.

How would you ensure immutability when you design classes that will be used in multi-threaded programming?

final fields, final class, no setters, no leaking mutable references.

Can we have mutable fields inside an object that is otherwise designed to be immutable?

Technically you can hold mutable objects, but then you must not expose them or you break true immutability.

What is the difference between the final keyword and immutability in Java?

final List variable cannot point to another list, but the list contents can still change. Immutable object design forbids changing internal state after creation.

14. Why is String immutable in Java?

Why is String immutable in Java?

Main reasons: (1) Security — strings used in paths, URLs, logins should not be silently changed. (2) Hash maps/sets — keys need stable hashCode; changing content would break lookups. 16. Deep copy vs shallow copy?

What is the difference between deep copy and shallow copy?

Shallow copy: New outer object; nested objects are copied by reference — inner object is still shared, so a change inside affects both.

Deep copy: Copies nested objects too — original and copy are fully independent.

Real life: Shallow: two roommates share one fridge. Deep: each apartment has its own fridge.

Explain the difference between String, StringBuilder, and StringBuffer.

What is the static keyword in Java?

24. Method overloading vs method overriding

What is the difference between method overloading and method overriding?

Overloading: Same method name, different parameter lists in one class — compiler picks at compile time (compile-time polymorphism).

Overriding: Subclass replaces parent’s instance method with the same signature — JVM picks at runtime (runtime polymorphism).

25. What are the rules for overriding a method in Java?

What are the rules for overriding a method in Java?

Only inherited methods can be overridden (not private). final and static methods cannot be overridden. Same name and parameters; return type must be the same or a subtype (covariant). Access cannot be more restrictive. Must not throw new or broader checked exceptions. Use super.method() to call parent version.

Indirect (follow-up) questions

Why can't private methods, static methods, or constructors be overridden in a subclass?

Private is not visible outside the class; static binds to class, not polymorphic instance dispatch; constructors are not inherited.

What happens if you try to override a final method in a subclass?

Compile error.

Can a subclass provide its own static method with the same signature as a static method in the superclass?

Yes — that is method hiding, not overriding.

What if subclass method has same name but different parameters than the parent?

That is overloading in the subclass context, not overriding.

Does changing parameter types in the subclass still count as overriding?

No — signature must match exactly for override.

How does Java tell overloading from overriding in a subclass?

Same signature as parent → override. Same name, different params → overload.

Can a public method in a parent class be overridden as protected or private in the child class?

No — you cannot reduce visibility.

Can a protected method in a parent class be overridden as public in the child class?

Yes — widening access is allowed.

Is it possible for an overriding method to have a completely different return type than the method in the parent class?

No — must be same type or subtype (covariant return).

How does covariant return type apply when you override a method in Java?

Child may return a more specific class than the parent method’s return type.

How can you call a parent class method from inside a method in the child class?

super.methodName()

27. What are this and super in Java? How are they used?

What are this and super in Java? How are they used?

this refers to the current object — disambiguate field vs parameter, pass “this” object around, or call this(...) constructor as first line.

super refers to the parent part — super(...) calls parent constructor (first line), super.field / super.method() access parent members when overridden or hidden.

You cannot have both this() and super() as non-first lines; only one chained call as the first statement in a constructor.

Indirect (follow-up) questions

In what situations would you need to use the this keyword inside a constructor?

When parameter names match field names — this.name = name makes the assignment clear.

Can this be used to call a method or access an instance variable in Java?

Yes — for instance members and to pass the current object.

Does this always refer to the current instance of the class you are in?

Yes — the object running the code right now.

super() to delegate to parent constructor; this() to delegate to another constructor in the same class.

28. Explain abstract classes in Java

Explain abstract classes in Java.

You cannot instantiate an abstract class directly. It can mix abstract methods (no body) and normal methods. Subclasses must implement abstract methods unless they are abstract too. Constructors are allowed and run when a concrete subclass is created.

Real life: “Payment” in an app — shared steps; each method (card, UPI) fills in details.

How can we achieve abstraction in Java?

Abstract classes and interfaces hide detail and expose only what callers need.

ATM — buttons for withdraw/balance; you do not see the bank’s internal wiring.

Are there any built-in abstract classes in the Java standard library?

Examples include AbstractList, InputStream, Reader.

No — but you can use AbstractType ref = new ConcreteType();.

Can an abstract class have a constructor? If yes, when does that constructor run?

Yes — used when subclasses are constructed to init shared parent state.

Can we declare an abstract method as static or private in Java?

No — abstract must be implemented by a subclass in the normal overriding sense.

Can an abstract class extend another abstract class?

Yes.

Yes — concrete subclasses must finish any unimplemented methods.

29. Explain interfaces in Java

Explain interfaces in Java.

An interface is a contract: implementing classes must provide the listed methods. Fields are public static final by default. Before Java 8, methods were abstract by default; Java 8 added default and static methods. No constructors. A class can implement many interfaces.

Real life: “Every payment provider must implement pay and refund” — your app talks to the interface, not each vendor’s internals.

Indirect (follow-up) questions

Can a class implement multiple interfaces in Java?

Yes — implements A, B, C.

If two interfaces declare methods with the same signature, or conflicting default methods, what must the implementing class do?

One implementation in the class satisfies both. If both define the same default method, the class must override and resolve the conflict.

Can an interface have a constructor in Java?

No — interfaces are not instantiated.

What is the difference between abstract class and interface if both can have abstract methods?

Abstract class can hold instance state and constructors; single inheritance. Interface is mainly contract + constants; multiple interfaces allowed.

Have you used any built-in Java interfaces in real projects? Can you name a few common ones?

Common ones: Runnable, Comparable, Serializable, AutoCloseable.

What happens if a concrete (non-abstract) class does not implement every method required by an interface it claims to implement?

The class must be declared abstract, or you get a compile error.

Can we define variables inside an interface, and if so, what modifiers do those variables have by default?

Yes — they are public static final constants, not instance fields.

Can interface methods be declared private or protected in Java, and how did that change in newer Java versions?

Classic interface methods are public. Since Java 9, private interface methods exist for helper code inside the interface. Protected is not used for interface methods.

Can an interface extend another interface, and can it extend more than one?

Yes — interfaces can extend one or more interfaces.

Why were default methods introduced in interfaces starting with Java 8?

To add new methods to interfaces without breaking existing implementations (backward compatibility).

Spring 1: What is Spring Boot?

What is Spring Boot?

How I’d answer (big picture): Think of Spring as a big toolkit for enterprise Java (dependency injection, web, data). Spring Boot is a convention-over-configuration layer on top: it tries to give you sensible defaults so you are not writing XML and glue code all day. If you are new to this topic, remember: Boot does not replace Spring—it makes Spring projects faster to start and easier to run in production.

  • Auto-configuration: If a library (e.g. Hibernate, Jackson) is on the classpath, Boot can wire common beans for you based on sensible defaults (you can still override).
  • Embedded server: You usually ship a JAR that already contains Tomcat/Jetty/Undertow—no separate app server install for simple deployments.
  • Starters: One Maven/Gradle dependency (e.g. spring-boot-starter-web) pulls a compatible stack of libraries so you are not hunting versions manually.
  • Why interviewers ask: They want to know you can deliver features quickly and operate the app (metrics, health, external config)—not only write beans.

Example (commented — good for whiteboard or follow-up):

@SpringBootApplication  // = @Configuration + @EnableAutoConfiguration + @ComponentScan
public class MyApplication {
    public static void main(String[] args) {
        // Boots the whole context and embedded server in one call
        SpringApplication.run(MyApplication.class, args);
    }
}

Indirect (follow-up) questions

What's new in Spring Boot 3.x?

Answering this: Boot 3 is a major upgrade—often asked so teams know you can migrate and run on current JDKs.

  • javax → jakarta: Package rename (e.g. javax.persistence → jakarta.persistence) because EE APIs moved under the Eclipse Foundation.
  • Java 17+: Baseline LTS; expect questions on records, sealed classes only if the role needs deep Java knowledge.
  • Native images: Better GraalVM story—smaller footprint and faster cold start for Kubernetes / cloud.

Can we create a non-web application in Spring Boot?

Answering this: Yes. “Boot” sounds web-only, but the web stack is optional; many teams use Boot for batch jobs and integration services.

  • When it makes sense: Scheduled jobs, message consumers, CLI tools, or workers that talk to queues/DB but expose no HTTP.
  • Typical hooks: Implement CommandLineRunner or ApplicationRunner to execute logic once the context is ready.
  • Practical note: You may exclude or avoid web starters if you truly do not need MVC—keeps the classpath lighter.

What are the advantages of Spring Boot?

How I’d answer (big picture): I frame this as developer speed plus operational readiness. Boot is popular because it reduces ceremony while still letting you drop down to plain Spring when needed.

  • Less boilerplate: Fewer XML files and manual bean wiring for common stacks (web, JPA, security).
  • Faster onboarding: Starters + auto-config mean new teammates can run the app locally quickly.
  • Production features: Actuator gives health, metrics, and info endpoints—useful for Kubernetes liveness/readiness and monitoring.
  • Externalized configuration: Properties/YAML + profiles so the same artifact runs in dev/stage/prod with different URLs and secrets.
  • DevTools (optional): Faster feedback during development (e.g. automatic restart on classpath changes).

Spring 3: What are embedded containers / servers? Default? How to add Jetty?

What are embedded containers / servers? Default? How to add Jetty?

How I’d answer (big picture): An embedded servlet container runs in the same process as your application. Classically you installed Tomcat separately and dropped a WAR in webapps; with Boot you often build a “fat” JAR that bundles the server, so deployment is “run this one file.”

  • Default in Boot: Tomcat is the default embedded server when you use spring-boot-starter-web.
  • Why it matters in interviews: Same artifact runs everywhere; ops teams like repeatable deploys and smaller moving parts for simple services.
  • Switching to Jetty: Exclude Tomcat from the web starter and add the Jetty starter so the classpath only has one embedded engine.
  • Trade-off: You still tune threads and timeouts—embedding does not remove the need to understand HTTP server behavior under load.

  <artifactId>spring-boot-starter-web</artifactId>

    <exclusion>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-tomcat</artifactId> <!-- remove default -->
    </exclusion>
  </exclusions>
</dependency>
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-jetty</artifactId> <!-- embed Jetty instead -->
</dependency>

Spring 4: Difference between Spring and Spring Boot?

Difference between Spring and Spring Boot?

How I’d answer (big picture): This question checks whether you know what is the framework vs what is the accelerator. Spring is the programming model; Boot is how most teams package and run Spring apps today.

  • Spring Framework: Core concepts—IoC container, DI, AOP, MVC, transaction abstraction, JDBC/JPA integration—you can use it with a lot of manual configuration.
  • Spring Boot: Adds starters, auto-configuration, embedded server defaults, and executable JAR layout so the happy path needs less setup.
  • Mental model: Your @Service and @RestController code is still “Spring”; Boot mainly decides which supporting beans to create and how to start the app.
  • When plain Spring still appears: Legacy apps, libraries, or teams that need full control over every bean definition.

Example (commented — good for whiteboard or follow-up):

Spring 5: Explain Spring MVC architecture

Explain Spring MVC architecture?

How I’d answer (big picture): MVC splits concerns: Model (data + domain rules), View (HTML/template for browsers), Controller (HTTP entry). In Spring, the central piece is DispatcherServlet—it is the front controller that dispatches work to your handlers.

  • Request flow: HTTP request → DispatcherServlet → HandlerMapping picks the right controller method → controller calls services → result goes to a view or is serialized.
  • REST vs server-rendered: With @RestController, return values become JSON/XML via message converters (no JSP by default). With Thymeleaf/JSP, a ViewResolver maps logical view names to templates.
  • Why it is testable: Controllers stay thin; heavy logic sits in services, so you can unit test without servlet containers when you design for it.
  • Interview tip: Mention DispatcherServlet + mapping + (optional) view resolution—shows you know the pipeline, not only annotations.

Example (commented — good for whiteboard or follow-up):

@RestController  // Controller + @ResponseBody on methods
@RequestMapping("/api/users")
public class UserController {

    @GetMapping("/{id}")
    public UserDto getUser(@PathVariable Long id) {
        // Service/repository called here — return object becomes JSON
        return userService.findById(id);
    }
}

How I’d answer (big picture): @SpringBootApplication is a composed annotation—one line on your main class that turns on the three things Boot apps usually need.

Spring 7: What is Dependency Injection?

What is Dependency Injection?

How I’d answer (big picture): Dependency Injection (DI) means an object’s collaborators are provided from outside instead of the object doing new ConcreteService() internally. In Spring, the IoC container creates singleton (by default) beans and wires them where constructors or setters ask for them.

  • Why it helps: You depend on interfaces or abstractions, so you can swap implementations (e.g. mock in tests, real in prod).
  • Constructor injection (preferred): Dependencies are explicit in the constructor signature—hard to forget and easy to make fields final.
  • Connection to IoC: Inversion of Control = the framework controls object creation; DI is the main mechanism you see in application code.
  • Interview phrase: “Spring manages the lifecycle; I declare what I need and the container satisfies it.”

Example (commented — good for whiteboard or follow-up):

@Component
public class Engine { }

@Component
public class Car {
    private final Engine engine;

    // Constructor injection — preferred in interviews
    public Car(Engine engine) {
        this.engine = engine;  // Spring passes a managed Engine bean
    }
}

Indirect (follow-up) questions

How does Spring achieve Inversion of Control (IoC)?

Answering this: IoC is the idea; DI is the mechanism Spring uses most of the time.

  • The container owns bean creation and wiring—you request collaborators via constructor/setter/field injection.
  • You can still call getBean manually, but in apps you usually rely on injection for clarity.

How does Spring Boot make DI easier than older Spring MVC setups?

Answering this: Boot does not change the DI model; it reduces how much you configure before DI works.

  • Component scanning finds your @Service classes without listing every bean in XML.
  • Auto-configuration registers infrastructure beans (e.g. DataSource) when properties and drivers are present.

Spring 8: What is IoC Container?

What is IoC Container?

How I’d answer (big picture): The IoC container is the runtime that holds your Spring beans: it instantiates them (often as singletons), injects dependencies, and applies lifecycle callbacks. In Spring Boot you almost always interact with an ApplicationContext subclass.

  • What you tell interviewers: “I don’t own object graphs for core services—the container does, and I declare beans or annotations.”
  • BeanFactory vs ApplicationContext: BeanFactory is the minimal SPI; ApplicationContext adds integration features (events, resource loading, easier annotation processing)—Boot defaults to ApplicationContext.
  • Lifecycle: Beans can implement InitializingBean/DisposableBean or use @PostConstruct/@PreDestroy for init/shutdown hooks.
  • First-time context: Think of the container as a registry + factory for your application’s collaborating objects.

Example (commented — good for whiteboard or follow-up):

// Rarely needed in app code (prefer @Autowired), but shows the idea:
@SpringBootApplication
public class DemoApplication {
    public static void main(String[] args) {
        ApplicationContext ctx = SpringApplication.run(DemoApplication.class, args);
        Car car = ctx.getBean(Car.class);  // container hands you a fully wired Car
        car.drive();
    }
}

Spring 9: Difference between @Component, @Service, and @Repository?

What is the difference between @Component, @Service, and @Repository?

How I’d answer (big picture): At runtime, all three stereotypes end up as Spring beans discovered by component scanning. The difference is mainly documentation and a little extra behavior, not a different registration mechanism.

  • @Component: Generic stereotype for any injectable helper (not tied to a layer name).
  • @Service: Convention for the business / application layer—signals “domain logic lives here” to humans and static analysis tools.
  • @Repository: Marks persistence components; Spring adds exception translation so low-level SQL exceptions become consistent DataAccessException types.
  • Practical guidance: Use the most specific stereotype that matches the layer so code reviews and onboarding stay clear.

Example (commented — good for whiteboard or follow-up):

@Service
public class OrderService { /* business rules */ }

@Repository
public class OrderJdbcDao { /* manual SQL — @Repository helps translate SQLException */ }

@Component
public class EmailNotifier { /* generic helper bean */ }

Indirect (follow-up) questions

Why separate controller, service, and DAO layers?

Answering this: Layering is about separation of concerns and testability.

  • Controller: HTTP mapping, validation binding, status codes—thin.
  • Service: Business rules, transactions, orchestration across repositories.
  • Repository/DAO: Persistence details—SQL or ORM calls.
  • Benefit: Change the database or API shape with less ripple; unit-test services without servlet APIs.

Is @Repository required with Spring Data JPA?

Answering this: Spring Data JPA repository interfaces are special: the framework generates implementations.

  • You usually extend JpaRepository and Spring Data registers the proxy—no class-level @Repository needed on the interface.
  • For hand-written JDBC DAO classes, @Repository is still valuable for exception translation.

Spring 10: What is application.properties?

What is application.properties?

How I’d answer (big picture): application.properties (or application.yml) is where you put external configuration: server port, datasource URLs, logging levels, feature toggles. The key idea is 12-factor style config—behavior changes without rebuilding the JAR.

  • Profiles: Use application-<profile>.properties and activate with spring.profiles.active so dev and prod differ cleanly.
  • Secrets: Prefer environment variables or a secret manager in production; properties files can reference env vars with ${VAR:default} syntax.
  • Type-safe config: @ConfigurationProperties maps a prefix to a Java class for validated, structured settings.
  • YAML vs properties: YAML is nicer for nesting; properties are simple and explicit—teams often pick one and stay consistent.
# application.properties — interview-friendly sample
server.port=8081
spring.datasource.url=jdbc:mysql://localhost:3306/mydb
spring.datasource.username=root
spring.datasource.password=${DB_PASSWORD:changeme}  # env override in prod

Indirect (follow-up) questions

How do you use different databases per environment?

Answering this: You never hardcode prod credentials in source; you switch profile-specific files or env-specific overrides.

  • Create application-dev.properties and application-prod.properties with different spring.datasource.* values.
  • Set SPRING_PROFILES_ACTIVE=prod (or JVM flag) in the deployment environment.
  • Use @Profile on @Bean methods when whole beans differ between environments (e.g. mock vs real payment client).

What is relaxed binding?

Answering this: Relaxed binding means Spring accepts several spellings for the same property key.

  • Examples: my.app.server-url, MY_APP_SERVER_URL, myAppServerUrl can map to one field in @ConfigurationProperties.
  • This reduces friction when mixing env vars (often UPPER_SNAKE) with property files (often kebab-case).

Spring 12: Types of Dependency Injection?

How many ways can dependency injection be done? / Types of DI?

How I’d answer (big picture): Spring supports three injection styles. In interviews, lead with constructor injection and mention the others as trade-offs.

  • Constructor injection: Dependencies declared in the constructor—best for required collaborators, immutable fields, and obvious APIs.
  • Setter injection: Optional or late-bound dependencies; risk is an object can be used before all setters run.
  • Field injection: @Autowired on fields—convenient for demos but hides dependencies and complicates unit tests.

Example (commented — good for whiteboard or follow-up):

@Service
public class OrderService {

    private final OrderRepository orders;
    private final PaymentClient payments;

    // Constructor injection — tell interviewer: "all deps visible + final fields"
    public OrderService(OrderRepository orders, PaymentClient payments) {
        this.orders = orders;
        this.payments = payments;
    }
}

Indirect (follow-up) questions

Why is setter injection risky for required deps?

Answering this: A required dependency should not be optional at runtime.

  • If a setter is forgotten, the bean still exists but fails later with NullPointerException or invalid state.
  • Constructor injection fails fast at startup if something is missing.

When might field injection be used?

Answering this: Mostly quick demos or legacy codebases; not the pattern I recommend for new services.

  • Harder to see required deps at a glance.
  • Testing often needs reflection or Spring’s test context instead of plain new.

Why prefer constructor injection?

Answering this: Teams and frameworks (including Spring) push constructor injection as the default good practice.

  • Makes dependencies explicit in the signature.
  • Allows final fields for immutability.
  • Easy to instantiate in tests: new Service(mockRepo).

Many dependencies (7–10)—what to do?

Answering this: A huge constructor is a design smell—the class may be doing too much.

  • Split responsibilities into smaller collaborators.
  • Introduce a facade or aggregate bean that groups related deps.
  • Consider configuration objects (@ConfigurationProperties) for many primitive flags.

Spring 13: Annotation-based container configuration?

What do you mean by annotation-based container configuration?

How I’d answer (big picture): Annotation-based configuration means you register and wire beans using Java annotations instead of (or alongside) XML. Most modern Spring Boot apps rely on this style.

  • Component scanning: Classes annotated with @Component, @Service, @Repository, @Controller become beans when they sit under the scanned packages.
  • Java config: @Configuration classes expose @Bean factory methods—useful for third-party classes you cannot annotate.
  • Wiring: @Autowired, constructor injection, @Value for single properties—together they replace large applicationContext.xml files.
  • First-time reader: XML is still supported, but annotations + Boot auto-config are the default mental model today.

Example (commented — good for whiteboard or follow-up):

@Configuration
public class AppConfig {
    @Bean
    public RestTemplate restTemplate() {
        return new RestTemplate();  // third-party class → @Bean factory method
    }
}

Spring 14: Explain @Autowired

Explain @Autowired

How I’d answer (big picture): @Autowired marks a dependency that the Spring container should satisfy by finding a matching bean (by type, then by name).

  • Where it applies: Constructor parameters, setter methods, or fields.
  • Single constructor rule: Since Spring Framework 4.3, if there is only one constructor, it is autowired automatically—@Autowired is optional.
  • Ambiguity: Multiple beans of the same type require @Qualifier or @Primary (covered in the next card).
  • Concept: You are declaring “this object needs a collaborator; please inject the right implementation.”

Example (commented — good for whiteboard or follow-up):

@Service
public class ReportService {
    private final UserRepository users;

    // @Autowired optional here — single constructor
    public ReportService(UserRepository users) {
        this.users = users;
    }
}

Indirect (follow-up) questions

How to wire beans without @Autowired?

Answering this: Manual wiring still exists for special cases.

  • @Bean methods that explicitly pass dependencies: return new X(yBean).
  • Legacy XML <constructor-arg> / <property> configuration.

Is @Autowired mandatory with a single constructor?

Answering this: No—optional on a single constructor for injection.

  • Spring will use that constructor to create the bean and resolve parameters from the context.

Spring 15: Explain @Qualifier and @Primary

Explain @Qualifier and @Primary

How I’d answer (big picture): When more than one bean matches an injection point’s type, Spring throws NoUniqueBeanDefinitionException unless you guide it.

  • @Qualifier("name"): Selects the bean whose qualifier or default name matches—most precise control at the injection site.
  • @Primary: Marks the default implementation when multiple exist and no qualifier is given—good for a main implementation plus alternates.
  • Precedence: An explicit @Qualifier at the injection point overrides @Primary when both apply.
  • Refactoring tip: Sometimes the cleaner fix is splitting interfaces so ambiguity disappears.

Example (commented — good for whiteboard or follow-up):

@Component("stripe")
@Primary
public class StripePayment implements PaymentService { }

@Component("paypal")
public class PayPalPayment implements PaymentService { }

@Service
public class CheckoutService {
    public CheckoutService(@Qualifier("paypal") PaymentService payment) {
        // Qualifier overrides @Primary — injects PayPal bean
    }
}

Indirect (follow-up) questions

Multiple beans same type—how to fix?

Answering this: Disambiguate or reduce the number of candidates.

  • Use @Qualifier on the injection point.
  • Mark one bean @Primary for the common case.
  • Or redesign so only one bean implements that narrow interface.

@Primary + @Qualifier together—which wins?

Answering this: @Qualifier wins because it is an explicit choice at the call site.

  • More specific injection-site instruction beats the default primary bean.

NoUniqueBeanDefinitionException—how to resolve?

Answering this: Same as ambiguity: Spring found multiple beans and needs a tie-breaker.

  • Add @Qualifier or @Primary, or remove duplicate registrations.

Only one bean—do @Primary / @Qualifier matter?

Answering this: Usually not—there is nothing to disambiguate.

  • They become important when you introduce a second implementation (e.g. feature flags, A/B clients).

Spring 16: application.properties vs application.yml?

What is the difference between application.properties and application.yml?

How I’d answer (big picture): Both files load the same Spring Environment; the difference is syntax and readability, not capability.

  • Properties: Simple key=value lines; great for flat config and easy diffs in code review.
  • YAML: Nested structure with indentation; good when many keys share a prefix (e.g. spring.datasource.*).
  • Pitfall with YAML: Indentation errors break parsing—some teams ban YAML for that reason.
  • Rule of thumb: Pick one style per project and avoid mixing two primary files without a clear convention.

Indirect (follow-up) questions

Spring 21: Difference between @Bean vs @Configuration

Difference between @Bean vs @Configuration

How I’d answer (big picture): @Configuration marks a class that contains @Bean factory methods. The important detail is full vs lite configuration mode.

  • @Configuration (full): Spring subclasses your config with CGLIB so inter-@Bean calls go through the container—singletons stay singletons.
  • @Configuration(proxyBeanMethods = false) or @Bean inside @Component: Inter-bean calls are plain Java—calling dataSource() twice can create two instances.
  • Interview takeaway: “@Configuration + @Bean methods cooperate like a registry; plain calls don’t.”

Example (commented — good for whiteboard or follow-up):

@Configuration
public class AppConfig {
    @Bean
    public UserRepository userRepository(DataSource ds) {
        return new UserRepository(ds);  // Spring intercepts: reuses singleton DataSource bean
    }
    @Bean
    public DataSource dataSource() { return DataSourceBuilder.create().build(); }
}

Indirect (follow-up) questions

Why multiple instances without @Configuration?

Answering this: Without the enhanced config proxy, method calls are direct invocations.

  • userRepository() calling dataSource() runs the real method body again instead of asking the context for the bean.
  • Result: multiple DataSource or repository instances that should have been singletons.

Spring 23: What is a Spring Bean?

What is a Spring Bean?

How I’d answer (big picture): A Spring bean is an object that the IoC container instantiates, configures, and manages (scope, lifecycle callbacks). It is not the same as “any Java object.”

  • DTO (Data Transfer Object): Carries data across layers or over the wire; fields may differ from the entity to avoid leaking internals.

POJO/DTO vs Spring bean?

Answering this: A class can be a POJO/DTO and also a bean if registered; many DTOs are not beans (e.g. JSON payloads created per request).

  • Entities (@Entity) are often beans when managed via repositories; API request bodies might just be deserialized POJOs.

NoSuchBeanDefinitionException though bean exists?

Answering this: The container truly does not see a bean definition matching your injection point.

  • Wrong package for component scan (bean in unscanned package).
  • Profile/condition not active, so the bean was never registered.
  • Typo in @Qualifier name or multiple candidates unresolved.

Spring 24: What are bean scopes?

What are bean scopes?

How I’d answer (big picture): Scope defines lifecycle and cardinality: how many instances exist and when they are created/destroyed.

  • singleton (default): One instance per Spring container—shared by all injectors.
  • prototype: New instance every time the bean is requested or injected (stateful objects—use carefully).
  • request / session: In web apps, one instance per HTTP request or user session—requires a web-aware context.
  • Interview note: Wrong scope causes subtle bugs (shared mutable state vs per-request isolation).

Indirect (follow-up) questions

Many components inject same service—same object?

Answering this: For singleton scope, yes—they share the same instance.

  • That is why services must be thread-safe if they hold mutable fields.

Prototype injected into singleton?

Answering this: A singleton only gets one prototype instance at injection time unless you use a lookup pattern.

  • Use ObjectFactory<T>, Provider<T>, or @Scope(value = ConfigurableBeanFactory.SCOPE_PROTOTYPE, proxyMode = ScopedProxyMode.TARGET_CLASS) when you need a fresh prototype per use.

What is the bean life cycle?

}

Spring 29: How to handle circular dependencies?

How to handle circular dependencies?

How I’d answer (big picture): A circular dependency means bean A needs B and B needs A (possibly through a longer chain). Spring cannot construct both with constructor injection without help.

  • Why constructors break: Each constructor waits for the other bean to exist first—deadlock at creation time.
  • Quick mitigations: @Lazy on one injection point injects a proxy that defers creation; setter injection breaks the cycle but is weaker for required deps.
  • Best fix: Extract shared behavior to a third service or invert the dependency direction so the graph is acyclic.
  • Team reality: Use @Lazy sparingly—treat it as a signal to refactor when you have time.

Example (commented — good for whiteboard or follow-up):

// Break cycle with @Lazy on one constructor parameter
@Service
public class A {
    public A(@Lazy B b) { }  // B resolved lazily — breaks circular ctor chain
}
@Service
public class B {
    public B(A a) { }
}

Spring 30: Eager vs lazy loading of beans?

What is eager loading and lazy loading of beans?

Spring 38: Best practices for entities?

Spring 39: JPA relationship types and @OneToOne example

What are the different types of relationships in JPA and explain @OneToOne with example

How I’d answer (big picture): JPA models cardinality between entities: one-to-one, many-to-one, one-to-many, many-to-many.

  • One-to-one: Each row on side A relates to at most one row on side B—often modeled with a shared primary key or a foreign key column on one side.
  • Owning side: Holds @JoinColumn (the FK). Inverse side: Uses mappedBy to reference the field name on the owner—prevents duplicate FK columns.
  • First-time reader: Think “one side owns the database column; the other mirrors the link logically.”

Example (commented — good for whiteboard or follow-up):

@Entity
class Person {
    @Id Long id;
    @OneToOne
    @JoinColumn(name = "passport_id")  // FK lives here — owning side
    Passport passport;
}
@Entity
class Passport {
    @Id Long id;
    @OneToOne(mappedBy = "passport")   // inverse — no extra FK column
    Person person;
}

Spring 40: Explain @OneToMany / @ManyToOne

Explain @OneToMany or @ManyToOne with example

How I’d answer (big picture): This is the most common mapping: parent/child or aggregate roots with children.

  • @ManyToOne on the child: Many rows reference one parent—FK lives on the child table (e.g. dept_id on employee).
  • @OneToMany on the parent: Collection navigation from parent to children; use mappedBy pointing to the child’s field that references the parent.
  • Cascade & orphan removal: Decide explicitly—cascade PERSIST helps when saving graphs; orphan removal deletes children removed from the collection.

Example (commented — good for whiteboard or follow-up):

@Entity
class Employee {
    @Id Long id;
    @ManyToOne
    @JoinColumn(name = "dept_id")  // FK on employee row
    Department department;
}
@Entity
class Department {
    @Id Long id;
    @OneToMany(mappedBy = "department")
    List<Employee> employees = new ArrayList<>();
}

Explain @ManyToMany with example

How I’d answer (big picture): Many-to-many means rows on both sides can reference multiple partners—relational databases model that with an association (join) table.

  • @JoinTable: Names the link table and join columns on the owning side.
  • Inverse side: Uses mappedBy to avoid duplicate join metadata.
  • Reality: Some teams model many-to-many as two one-to-manys with an explicit link entity for extra columns (e.g. role since date).

Employee has many managers and manager has many employees—how many tables?

Answering this: Typically three physical tables: two entity tables plus a join table holding pairs of keys.

  • If you need attributes on the relationship itself, promote the join row to its own entity.

Spring 42: What is cascading in JPA?

What is cascading in JPA and when would you use it?

  • Common types: PERSIST, MERGE, REMOVE, REFRESH, DETACH, or ALL as shorthand.
  • When to use: Aggregate patterns—child rows only make sense with the parent (order line items).

Spring 43: Eager vs lazy loading in JPA?

How does eager loading and lazy loading work in JPA?

How I’d answer (big picture): Fetching controls when related data is loaded from the database relative to the root entity.

  • EAGER: Association loaded with the parent—simpler mental model but can over-fetch graphs.
  • LAZY (collections/associations default in many cases): Proxies load data on first access within an active persistence context.
  • LazyInitializationException: Touching a lazy association with no open session—common when returning entities directly to a web layer outside a transaction.
  • Fixes: JOIN FETCH in JPQL, entity graphs, DTO projections, or @Transactional on the service boundary that still materializes data before serialization.

LazyInitializationException?

Answering this: The proxy tried to load data but the persistence context was already closed.

  • Fetch eagerly in the query, or keep the session open for the whole use-case, or map to DTOs inside the transaction.

Spring 45: What is JPQL vs HQL?

What is JPQL and how is it different from HQL?

How I’d answer (big picture): JPQL (Java Persistence Query Language) is the standard JPA query language—object-oriented syntax that compiles to SQL for your database.

  • Portable: Targets entity names and fields, not necessarily physical table/column strings.
  • HQL (Hibernate Query Language): Hibernate’s native query language; largely overlaps JPQL but includes vendor extensions.
  • Rule of thumb: Prefer JPQL in portable code; drop to native SQL when you must optimize with database-specific features.

Spring 46: What is @Query?

What is @Query (custom queries) and why use them?

How I’d answer (big picture): @Query on a Spring Data repository method supplies JPQL or SQL explicitly instead of relying on method-name derivation.

  • When to use: Complex joins, fetch plans, aggregations, or when method names become unreadable.
  • Parameters: Named parameters with @Param keep queries readable and refactor-safe.
  • Native queries: Set nativeQuery = true for raw SQL—fastest path to DB-specific SQL but least portable.

    @Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.id = :id")

    List<Order> findRawByStatus(@Param("s") String status);
}

Problems with native queries?

Answering this: You trade portability and compile-time safety for control.

  • Dialect-specific functions may not work if you switch databases.
  • Schema renames break SQL strings—cover with tests and migrations.

Spring 47: What is @NamedQuery and @NamedNativeQuery?

Spring 48: What is the use of @Transactional?

What is the use of @Transactional?

How I’d answer (big picture): @Transactional declares that a method (or class) should run inside a database transaction managed by Spring’s transaction abstraction.

  • Atomicity: All participating persistence operations commit together or roll back on failure (default: rollback on unchecked exceptions).
  • Typical placement: Service layer methods that orchestrate multiple repository calls for one business operation.
  • Proxy mechanism: Spring wraps beans with a transactional proxy—external calls honor annotations; internal self-calls usually do not.

Example (commented — good for whiteboard or follow-up):

@Service
public class TransferService {
    private final AccountRepository accounts;

    @Transactional  // one transaction for both updates
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        Account from = accounts.findById(fromId).orElseThrow();
        Account to = accounts.findById(toId).orElseThrow();
        from.debit(amount);   // if anything fails below, both roll back
        to.credit(amount);
        accounts.save(from);
        accounts.save(to);
    }
}

Transactional method called from same class?

Answering this: Same-class calls skip the Spring proxy unless you restructure.

  • Extract to another bean, use ApplicationContext self-injection carefully, or AspectJ weaving for advanced cases.

Best layer for @Transactional?

Answering this: Services own business transactions; repositories stay focused on persistence.

  • Keeps transaction boundaries aligned with use-cases, not individual SQL statements.

Spring 49: What is propagation in @Transactional?

What is propagation in Transaction?

How I’d answer (big picture): Propagation defines how a new transactional method interacts with an existing transaction on the call stack.

  • REQUIRED (default): Join the caller’s transaction if present; otherwise create a new one.
  • REQUIRES_NEW: Always start a fresh transaction; suspends the outer one—use for audit logs that must persist independently.

Spring 51: What is the N+1 problem and how do you fix it?

What is the N+1 problem and how do you fix it?

How I’d answer (big picture): The N+1 query problem happens when you load a list of parent entities with one query, then touch a lazy collection or association for each row—triggering one extra query per parent. Total queries ≈ 1 + N instead of a small constant.

  • Classic JPA example: SELECT s FROM Student s returns 100 students; then s.getCourses().size() runs a query per student → 1 + 100 (or more if each access is naive).
  • Why it hurts: Latency and DB load scale with rows; in APIs listing hundreds of parents, this becomes a bottleneck.
  • Fix — JOIN FETCH: SELECT s FROM Student s JOIN FETCH s.courses loads students and courses in fewer round-trips (watch for duplicate rows and use DISTINCT where needed).
  • Alternatives: @EntityGraph / subgraphs, batch fetching (@BatchSize in Hibernate), DTO projections, or explicit native queries—pick based on read shape.

Example (commented — good for whiteboard or follow-up):

List<Student> students = em
    .createQuery("SELECT s FROM Student s", Student.class)
    .getResultList();
for (Student s : students) {
    s.getCourses().size(); // lazy -> extra SELECT per student (N+1)
}

// Fix pattern (repository):
@Query("SELECT DISTINCT s FROM Student s JOIN FETCH s.courses")
List<Student> findAllWithCourses();

Indirect (follow-up) questions

Why might the user’s example say 10,000 queries for 100 students?

Answering this: That number usually means another nested lazy access (e.g. per course) or a misunderstanding—the core idea is still “one query per iteration.”

  • Clarify in interview: N+1 means one per parent row for each lazy hop you trigger in a loop.
  • Profile with SQL logging (spring.jpa.show-sql, Hibernate statistics) to count statements.

Spring 52: Difference between JOIN and JOIN FETCH?

Difference between JOIN and JOIN FETCH in JPQL?

How I’d answer (big picture): In JPQL/HQL, a plain JOIN (or JOIN in the WHERE) is mainly for filtering or correlation—it does not by itself force Hibernate to populate the associated entity graph in memory. JOIN FETCH tells the provider to fetch the association in the same query (eager load along that path).

  • Use JOIN: “Students who have at least one course” — you only need the condition, not loading all Course entities.
  • Use JOIN FETCH: “Students and I will read courses in this request” — avoids lazy loads and N+1.
  • Caveat: FETCH joins can duplicate rows; use DISTINCT or DTO mapping when appropriate.

Example (commented — good for whiteboard or follow-up):

// Filter only — may still lazy-load courses later
@Query("SELECT s FROM Student s JOIN s.courses c WHERE c.name = :name")

// Load graph now
@Query("SELECT DISTINCT s FROM Student s JOIN FETCH s.courses")
List<Student> findAllWithCourses();

Spring 53: How does locking work in Spring JPA?

How does locking work in Spring JPA?

How I’d answer (big picture): Locking coordinates concurrent transactions so reads/writes don’t corrupt shared rows. In JPA you declare lock mode on queries or entity operations; Spring Data exposes @Lock on repository methods.

  • Optimistic locking: No DB row lock by default—use a @Version column; on update, if version changed, you get OptimisticLockException. Good when conflicts are rare.
  • Pessimistic locking: Database locks the row (PESSIMISTIC_READ / PESSIMISTIC_WRITE) so others block or fail per DB semantics—use when conflicts are likely (e.g. account balance).
  • Spring: @Lock(LockModeType.PESSIMISTIC_WRITE) on find... methods; ensure a transaction wraps the call.

Example (commented — good for whiteboard or follow-up):

@Entity
public class Product {
    @Id private Long id;
    private int stock;
    @Version
    private int version;
}

public interface ProductRepository extends JpaRepository<Product, Long> {
    @Lock(LockModeType.OPTIMISTIC)
    Optional<Product> findWithOptLock(Long id);

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    Optional<Product> findByNameForUpdate(String name);
}

Indirect (follow-up) questions

Optimistic vs pessimistic in one sentence?

Answering this: Optimistic = detect conflicts at commit with version; pessimistic = lock rows early so others wait.

  • E-commerce catalog read → often optimistic; bank transfer on one row → often pessimistic.

Spring 54: Explain JPA architecture

Explain JPA architecture

How I’d answer (big picture): JPA standardizes how Java objects map to relational data. At a high level: entities ↔ tables, EntityManager ↔ unit of work, persistence context ↔ first-level cache, EntityManagerFactory ↔ expensive factory created once.

  • Persistence / bootstrap: Persistence.createEntityManagerFactory("pu") (standalone); in Boot, auto-config builds the factory for you.
  • EntityManager: CRUD, queries, transactions (in JSE) — the API you use for a unit of work.
  • Persistence context: Managed entities and dirty tracking for that EntityManager session.
  • Provider (e.g. Hibernate): Implements JPA, generates SQL, talks JDBC to the DB.

em.close();

How does this map to Spring Data JPA?

Answering this: Your repository calls are implemented by a generated proxy that delegates to EntityManager (and the shared EntityManagerFactory under the hood).

Spring 55: What happens when you call repository.save()?

What happens under the hood when you call .save() on a Spring Data repository?

How I’d answer (big picture): save on SimpleJpaRepository decides whether the entity is new or existing, then either persist (insert) or merge (update) via the EntityManager. SQL typically flushes at transaction commit or on flush().

  • Proxy: Your interface is a Spring Data proxy at runtime—not your handwritten class.
  • Persistence context: Managed entities are tracked; changes dirty-check to UPDATE statements.
  • Transaction: Usually your service method is @Transactional; commit triggers flush.
  • Return value: The managed (possibly merged) entity instance.

Example (commented — good for whiteboard or follow-up):

@Service
public class UserService {
    private final UserRepository users;

    public User saveUser(User u) {
        return users.save(u); // -> SimpleJpaRepository.save -> em.persist/merge
    }
}

Indirect (follow-up) questions

save() vs saveAndFlush()?

Answering this: save() registers state; flush may wait until commit. saveAndFlush() forces a flush immediately so subsequent queries in the same transaction see DB state.

  • Use saveAndFlush when you need generated IDs or constraints validated before the method ends.

persist() vs merge()?

Answering this: persist — new entity, becomes managed. merge — take a detached (or copied) instance and attach state to the persistence context (update path).

  • save() uses heuristics (e.g. null id / Version / Persistable) to choose.

Spring 56: What is the persistence context?

What is the persistence context in JPA?

How I’d answer (big picture): The persistence context is the first-level cache: a set of managed entities for a given EntityManager (per transaction/thread in typical Spring setups). Repeat find by id returns the same instance; changes are tracked until flush.

  • Dirty checking: At flush, the provider compares snapshots and emits UPDATEs for changed fields.
  • Many updates, one commit: If you change a field 100 times in one transaction, only the final state matters for that column at flush—not 100 UPDATEs for each keystroke.
  • Scope: Cleared when the persistence context closes or clears—not a cluster-wide cache.

Example (commented — good for whiteboard or follow-up):

em.getTransaction().begin();
Employee e1 = em.find(Employee.class, 1L); // SQL
Employee e2 = em.find(Employee.class, 1L); // same persistence context, no second SELECT
e1.setName("John Updated");
em.getTransaction().commit(); // UPDATE issued

Indirect (follow-up) questions

100 name changes before commit—how many DB updates?

Answering this: Typically one UPDATE for that entity at flush (unless you flush repeatedly). Intermediate values are not persisted individually.

  • What matters is managed state at flush time vs loaded snapshot.

Spring 57: What are the different entity states?

What are the different lifecycle states of a JPA entity?

How I’d answer (big picture): JPA entities move between states relative to the persistence context.

  • Transient (new): new Entity() — not known to JPA; no DB row yet.
  • Managed (persistent): After persist or after loaded with find — tracked; updates flush to DB.
  • Detached: Was managed, but context closed or detach — changes not auto-synced until merge.
  • Removed: remove() scheduled DELETE on commit while still associated with a context.

Spring 58: How does caching work in JPA?

How does caching work in JPA?

How I’d answer (big picture): Two levels matter in interviews: first-level (persistence context, always on) and second-level (shared cache at EntityManagerFactory, optional, provider-specific).

  • L1: Per-session cache; guarantees identity within a transaction for managed entities.
  • L2: Cross-session read cache (e.g. EhCache, Infinispan) — must configure concurrency strategy; stale data is possible if DB changes outside JPA.
  • Invalidation: Hibernate updates cache on writes when configured; external DB edits need eviction/refresh/TTL.

Example (commented — good for whiteboard or follow-up):

@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)

Indirect (follow-up) questions

Can caching serve stale data?

Answering this: Yes—especially L2 or read-through caches if another node updates the DB or you bypass Hibernate.

  • Use strategies like READ_WRITE, TTL, or distributed invalidation in clusters.

Spring 69: What is JWT and how does it differ from Basic auth?

What is JWT and how does it differ from Basic authentication?

How I’d answer (big picture): JWT is a compact, signed (or MAC’d) token format—often used for stateless APIs. Basic auth sends username:password (Base64) on every request.

  • JWT pros: Server can validate signature + expiry without session store; good for microservices.
  • Basic cons: Password-like material on every call unless HTTPS; prefer short-lived tokens or sessions for browsers.
  • Sessions vs JWT: Sessions store state server-side; JWT pushes claims to the client—trade-offs on revocation and size.

Indirect (follow-up) questions

JWT vs sessions in microservices?

Answering this: JWT avoids every service hitting a shared session store; each validates the token with a shared secret or public key.

  • Downside: revocation and token size—often pair access tokens with refresh tokens or short TTL.

Spring 70: What is the structure of a JWT?

What is the structure of a JWT?

How I’d answer (big picture): Three Base64url-encoded parts separated by dots: header.payload.signature.

  • Header: Algorithm (alg) and type (typ).
  • Payload: Claims—sub, exp, roles, custom data (do not put secrets here—payload is readable).
  • Signature: Ensures integrity—HMAC with secret or RSA/ECDSA with key pair.

Answering this: Validation fails; respond 401. Client uses refresh token or re-login; avoid long synchronous work before checking expiry at request start.

  • Some teams issue short-lived access tokens + refresh endpoint.

Spring 90: What is AOP (aspect-oriented programming)?

What is AOP?

How I’d answer (big picture): AOP separates cross-cutting concerns (logging, transactions, security) from business code using aspects applied at join points (usually method execution).

  • In Spring: JDK dynamic proxies (interfaces) or CGLIB subclasses for concrete classes.
  • Examples: @Transactional, @Cacheable, custom @Aspect with @Before/@Around.

Indirect (follow-up) questions

When use AOP?

Answering this: Whenever behavior repeats across many methods and isn’t the core domain rule (logging, metrics, access control wrappers).

  • Keeps services focused on business logic.

Spring 91: What are the core components of AOP?

What are the core components of AOP?

How I’d answer (big picture): Aspect = module of cross-cutting code. Join point = point in execution (method call). Pointcut = expression selecting join points. Advice = code that runs (@Before, @After, @Around). Weaving = applying aspects (Spring does runtime proxy weaving).

  • @Around is most powerful: can skip/proceed/replace return value via ProceedingJoinPoint.proceed().

Spring 92: How do proxies work in Spring AOP?

How do proxies work in Spring AOP?

How I’d answer (big picture): Spring wraps beans in a proxy. Callers inject the proxy; advice runs, then the target method. Internal this.foo() calls skip the proxy—same self-invocation issue as transactions.

  • CGLIB vs JDK: Class-based proxy needs non-final methods; interface JDK proxies need interface methods.

Spring 93: Which classes or methods should not be final in Spring?

Which Spring features conflict with final classes or methods?

How I’d answer (big picture): CGLIB proxies subclass your bean—final classes/methods cannot be overridden, so transactional/cache/async aspects may not apply. Prefer non-final service classes or interface-based JDK proxies.

  • Affected: @Transactional, @Cacheable, @Async, @Configuration with full proxy bean methods.

Spring 94: What is the self-invocation problem?

What is the self-invocation problem in Spring?

How I’d answer (big picture): Calling this.otherMethod() inside the same class bypasses the proxy, so @Transactional, @Async, AOP aspects on otherMethod won’t run. Fix: move to another bean, inject self, or AopContext.currentProxy() (requires exposeProxy).

  • Best fix: Extract collaborator service—clearest design.

Example (commented — good for whiteboard or follow-up):

@Service
public class PaymentService {
    @Transactional
    public void process() { saveTx(); } // saveTx not transactional — self-invocation
    @Transactional
    public void saveTx() { }
}

How do you schedule tasks in Spring Boot?

Spring 96: fixedDelay vs fixedRate in @Scheduled

What is the difference between fixedDelay and fixedRate?

How I’d answer (big picture): fixedDelay waits X ms after the previous execution finishes before starting again. fixedRate tries to start every X ms from the schedule start—overlaps possible if work runs long.

  • Choose delay for back-pressure; rate for strict periodic sampling.

Spring 97: How do you run scheduled tasks in parallel?

How can scheduled tasks run in parallel?

How I’d answer (big picture): Default scheduler uses a single thread—long tasks block others. Provide a TaskScheduler @Bean (e.g. ThreadPoolTaskScheduler) with pool size > 1 so different @Scheduled methods can run concurrently.

  • Also consider: @Async for fire-and-forget work kicked from scheduled jobs.

Example (commented — good for whiteboard or follow-up):

@Bean
TaskScheduler taskScheduler() {
    ThreadPoolTaskScheduler s = new ThreadPoolTaskScheduler();
    s.setPoolSize(5);
    s.setThreadNamePrefix("sched-");
    return s;
}

Spring 98: How do you handle failures in scheduled tasks?

How do you handle scheduled task failures?

How I’d answer (big picture): Wrap logic in try/catch and log; configure a scheduler ErrorHandler for uncaught exceptions; add retries with Spring Retry for transient failures.

  • Monitoring: Alert on repeated failures—silent exceptions in background threads are easy to miss.

Spring 99: How do you run work in the background (@Async)?

How do you run a task in the background in Spring Boot?

How I’d answer (big picture): Enable @EnableAsync, annotate methods with @Async, and provide an Executor bean if you need a thread pool. Return CompletableFuture if the caller needs a result.

  • Use cases: emails, notifications, long I/O off the request thread.
  • Caveat: Self-invocation bypasses async proxy—same as transactions.

Indirect (follow-up) questions

CompletableFuture vs @Async?

Answering this: CompletableFuture is Java’s async composition API; @Async is Spring’s way to run a bean method on another thread via proxy.

  • You can return CompletableFuture from @Async methods.

Spring 100: How does @Async work behind the scenes?

How does @Async work behind the scenes?

How I’d answer (big picture): Spring creates a proxy around your bean. Calls to @Async methods are submitted to a TaskExecutor (default simple executor or your ThreadPoolTaskExecutor). The caller thread returns immediately unless it waits on a Future.

  • Pool tuning: Define @Bean Executor named taskExecutor or use @Async("beanName").
  • Same-class calls are not async—must go through injected bean.

Example (commented — good for whiteboard or follow-up):

@Service
public class AsyncService {
    @Async
    public CompletableFuture<String> work() {
        return CompletableFuture.completedFuture("done");
    }
}

Java Scenario 1: ATM operations — switch

Write logic to handle ATM operations using Scanner input.

  • Ask the user for the operation type: withdraw, deposit, or check_balance.
  • withdraw: read the amount; if the balance is sufficient, deduct it and confirm; otherwise print Insufficient funds.
  • deposit: read the amount and add it to the balance.
  • check_balance: display the current balance.

Hint: Use a switch on the operation string and if-else (or a simple condition) to validate the balance on withdrawal.

import java.util.Scanner;

public class ATM {
    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);

        double balance = 1000; // initial balance

        System.out.println("Enter operation (withdraw, deposit, check_balance):");
        String operationType = sc.nextLine();

        switch (operationType) {

            case "withdraw":
                System.out.println("Enter amount to withdraw:");
                double withdrawal_amount = sc.nextDouble();

                if (balance >= withdrawal_amount) {
                    balance = balance - withdrawal_amount;
                    System.out.println("Withdrawal successful. Balance: " + balance);
                } else {
                    System.out.println("Insufficient funds");
                }
                break;

            case "deposit":
                System.out.println("Enter amount to deposit:");
                double deposit_amount = sc.nextDouble();

                balance = balance + deposit_amount;
                System.out.println("Deposit successful. Balance: " + balance);
                break;

            case "check_balance":
                System.out.println("Current balance: " + balance);
                break;

            default:
                System.out.println("Operation not found");
        }

        sc.close();
    }
}

Java Scenario 3: Password strength checker

/Java

Given a String password, loop through its characters and classify the password’s strength.

  • Count how many characters are uppercase, lowercase, digits, and special (anything that is not upper, lower, or digit).
  • If all four categories appear at least once and length ≥ 8, print Strong password; otherwise print Weak password.

Hint:

  • Loop through each character (index 0 … length() - 1).
  • Use Character.isUpperCase, Character.isLowerCase, and Character.isDigit for the first three kinds.
  • Treat any other character as special.
  • Strength: at least one from each category and length >= 8.
import java.util.Scanner;

public class PasswordStrengthChecker {
    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);
        System.out.println("Enter password:");
        String password = sc.nextLine();
        sc.close();

        int upper = 0, lower = 0, digit = 0, special = 0;

        for (int i = 0; i < password.length(); i++) {
            char c = password.charAt(i);
            if (Character.isUpperCase(c)) {
                upper++;
            } else if (Character.isLowerCase(c)) {
                lower++;
            } else if (Character.isDigit(c)) {
                digit++;
            } else {
                special++;
            }
        }

        boolean allCategories = upper > 0 && lower > 0 && digit > 0 && special > 0;
        boolean strong = allCategories && password.length() >= 8;

        System.out.println(strong ? "Strong password" : "Weak password");
    }
}

Java Scenario 6: Print even and odd numbers with two threads

/Java

Two threads should print the numbers 1 through 20. One thread prints only even numbers, the other only odd numbers, in the correct sequence.

Hint:

  • Use synchronization (e.g. a shared lock and wait() / notify(), or ReentrantLock with conditions) so the threads take turns and numbers appear in order 1, 2, …, 20.

Thread syntax (read this first)

Thread — A separate flow of execution. You give it work as a Runnable (code that should run on that thread).

new Thread(() -> { ... }) — The () -> { } is a lambda: shorthand for the run() method’s body. (You can also use an anonymous Runnable class with run() if you prefer.)

start() — Starts the thread; the JVM calls run() on that thread. Do not call run() yourself if you want real parallel execution.

join() — main (or any thread) can call t.join() to wait until thread t finishes, so the program doesn’t exit early.

Synchronization (for turn-taking)

  • synchronized (lock) { ... } — Only one thread at a time can execute that block for the same lock object.
  • lock.wait() — The current thread releases the lock and waits until another thread calls notify() on the same lock.
  • lock.notify() — Wakes one thread that is waiting on this lock.

Minimal example (no shared counter yet):

Thread t = new Thread(() -> {
    System.out.println("Hello from a thread");
});
t.start();
t.join(); // wait until t finishes

What synchronized, wait(), and notify() are doing

Picture one shared variable number and one “room key” lock. Only the thread holding the key may read or change number and print. That is what synchronized (lock) { … } enforces: at most one thread runs that block at a time for this same lock object.

The odd thread should only act when number is odd; the even thread only when number is even. If the wrong thread enters first, it must not print yet — it should wait.

lock.wait() means: “I give up the key, go to sleep, and let the other thread enter synchronized.” Without this, both threads could get stuck or fight forever. You may only call wait() while you are inside synchronized (lock) — you must be holding the key to hand it back.

lock.notify() means: “I’m waking up one thread that called wait() on this same lock.” After printing and doing number++, we notify() so the other thread can wake up, grab the lock, and take the next turn.

The inner while (not just if) repeats: wake up → check again → if it’s still not my turn, wait() again. That handles “spurious wakeups” and keeps ordering correct.

Flow in one line: enter synchronized → if not my parity, wait() → else print, number++, notify() → leave block → repeat.

Full solution (even/odd with wait / notify):

public class EvenOddPrint {

    private static final int MAX = 20;
    private static int number = 1;
    private static final Object lock = new Object();

    public static void main(String[] args) throws InterruptedException {

        Thread odd = new Thread(() -> {
            try {
                while (true) {
                    synchronized (lock) {
                        while (number <= MAX && number % 2 == 0) {
                            lock.wait();
                        }
                        if (number > MAX) {
                            lock.notify();
                            break;
                        }
                        System.out.println("Odd:  " + number);
                        number++;
                        lock.notify();
                    }
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }, "odd-thread");

        Thread even = new Thread(() -> {
            try {
                while (true) {
                    synchronized (lock) {
                        while (number <= MAX && number % 2 != 0) {
                            lock.wait();
                        }
                        if (number > MAX) {
                            lock.notify();
                            break;
                        }
                        System.out.println("Even: " + number);
                        number++;
                        lock.notify();
                    }
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }, "even-thread");

        odd.start();
        even.start();
        odd.join();
        even.join();
    }
}
