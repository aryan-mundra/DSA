1. Resume Metrics Interrogation

"You put numbers on paper. Numbers are promises. Prove them."

CORE You claim you "reduced recharge workflow latency by 55% (900ms → 400ms)" with a Redis caching strategy. Walk me through exactly what was slow, what you cached, and how you measured the 900ms and the 400ms.
SIMPLE ANSWER

The slow part was loading the recharge plans. One request took about 900ms. Most of that time was the database doing a big join to fetch plans, plus a call to another service for offers.

Those plans don't change often (only a few times a day), but lots of users ask for them. So they are perfect to cache. I saved the ready-made plan list in Redis (a fast in-memory store) with a key like plans:{circle}. Now most requests get the answer straight from Redis and skip the slow database work. That brought it down to about 400ms.

I measured both numbers from our monitoring tool (APM), comparing the same time of day before and after, so I wasn't comparing a busy hour to a quiet one.

55% off the p95? Or the average? Because those are very different claims. Which one, and what was the p99?

ANSWER

It was the p95 — that means 95% of requests were faster than this number. I use p95 because it shows what slow users actually feel, not just the lucky average. The p99 (the slowest 1%) improved less, around 30–35%, because those are mostly cache misses and other slow bits that caching alone can't fix.

A cache hit ratio of 90%. How did you actually compute that, and what did you do about the 10% misses that still hit Oracle during peak?

ANSWER

Redis tells you. It tracks hits and misses, and hit ratio = hits ÷ (hits + misses) — ours was about 90%. The 10% misses mostly happen right when a cached item expires and many users ask at once — they all rush to the database together (called a cache stampede). I fixed that two ways: add a small random time to expiry so they don't all expire together, and let only one request rebuild the cache while the others wait a moment.

You used a TTL. So your data is stale for the entire TTL window. On a payments-adjacent system, how do you justify serving a customer a plan price that might be wrong?

ANSWER

Important point: I only cache the plan list for showing on screen. The real price the customer actually pays is checked again from the database at payment time. So a slightly old list never causes a wrong charge — at worst the user sees a plan that just changed, and checkout corrects it. And when a price really must update instantly, the catalog service sends an event (over Kafka) and we delete that cached item right away. So the timer is the backup, and the event is the instant fix.

Why I'm asking: Most people can't tell the difference between "cache to be fast" and "money must be correct." If you had told me you cache the real price and charge from it, this part of the interview would be over — badly.
CORE "Cutting plan-fetch response times by 40%" on SQL/Oracle. 40% relative to what baseline, and how do I know it wasn't just because traffic was lower that week?
SIMPLE ANSWER

The "before" number came from the database's own reports and our slow-query logs, not from one random moment. The trick is I looked at how much work the query did per run (how many data blocks it read), not just the clock time. Work-per-run doesn't change when traffic changes — so when that dropped about 40%, it proves the query itself got cheaper, not that the week was quiet. The fix was adding the right index so the database could jump straight to the rows instead of scanning the whole table.

"Buffer gets per execution dropped." Good. But adding an index isn't free. What did that index cost you on the write path, and how did you decide it was worth it?

ANSWER

An index makes reads faster but writes a little slower, because every insert or update must also update the index. But these plan tables are read millions of times and changed only a few times a day. So faster reads easily win. I checked that writes to these tables were not on any busy path, so the small write cost didn't matter.

Show me the difference between an index range scan and a full table scan in the execution plan. What specific line tells you the optimizer chose your index?

ANSWER

You run EXPLAIN PLAN to see how the database will run the query. If it says TABLE ACCESS FULL, it is reading the whole table — slow. After my index, it says INDEX RANGE SCAN with my index name — that means it used the index to jump to just the rows it needs. I also check that the cost number went down.

Kill shot if you flinch: If someone says "I added an index and it got faster" but can't read the plan or name the work-per-run number, I assume they watched a teammate do it. The work-per-run metric is the proof.
DEEP "Reducing undetected production failures by 60%." How do you measure a failure you didn't detect before? That's a number about things you couldn't see.
SIMPLE ANSWER

Fair point. Before, we found out something broke only when a customer complained or another team noticed. After I added alerts (on error rate, slow responses, and a few key error types), the system told us first. So I counted: how many problems were first reported by a person, versus first caught by an alert? The "person found it first" cases dropped by about 60%. In short, we now catch problems faster, before users feel them.

What did you alert on? If you say "log errors," I'll ask why your pager isn't going off 4,000 times a night.

ANSWER

Not every error — that would wake people all night and they'd start ignoring it. I alerted on real signs of trouble: error rate going above normal for a while, responses getting too slow, and a few serious errors like the payment gateway being unreachable. A quick blip doesn't alert anyone; a real, lasting problem does. Dashboards are for looking; alerts are only for "a human must act now."

2. Java & JVM at Your Scale

"You run Java backends at 1.5M+ transactions/day. I only care about the JVM behavior that bites you at that traffic — not textbook trivia."

DEEP Your service handles 1.5M transactions/day. Walk me through a garbage collection pause causing a latency spike. Which collector, and how would you debug it?
SIMPLE ANSWER

Java cleans up unused memory automatically — that's garbage collection (GC). To do a big cleanup, Java sometimes pauses your whole app for a moment ("stop-the-world"). During that pause, no request is served, so users see a sudden slow spike.

For a busy service I use the G1 collector, which is built to keep pauses short, and I tell it a target pause time. To debug, I turn on GC logs and line up the pause times with the slow spikes in our monitoring. If pauses keep growing, it's usually a memory leak (we're holding objects we don't need). If pauses are frequent but memory is steady, we're creating too many objects too fast and the young area is too small.

You said "memory leak in Java." There's a garbage collector. How is a leak even possible?

ANSWER

Java only deletes objects that nothing is pointing to. A "leak" means we are still pointing to objects we no longer need, so Java thinks they're in use and keeps them. Example: a list marked static that we keep adding to and never clear. It's not Java's fault — it's our code holding on to stuff.

You mentioned ThreadLocal on a pooled thread. Explain that leak precisely — it's a classic in a Spring app on Tomcat.

ANSWER

Web servers reuse the same worker threads again and again. ThreadLocal stores a value on a thread. If I set a value during a request but forget to remove it, that value stays stuck on the thread forever (a leak), and the next request on that same thread might even see the old user's data (a bug). The fix: always clear it in a finally block — set it, use it, then remove() it.

3. Concurrency & Threading

"You wrote 'concurrency tuning' on your resume. That word is a magnet for me."

DEEP Your recharge service calls a payment gateway. You use a thread pool. The gateway gets slow. Walk me through exactly how your entire service dies, and how you'd prevent it.
SIMPLE ANSWER

Your service has a fixed set of worker threads (a thread pool). Normally a request to the payment gateway takes 50ms and the thread is free again quickly. Now the gateway slows to 5 seconds. Each request now holds its thread for 5 seconds. Soon all threads are stuck waiting, the queue fills up, and even requests that don't touch the gateway can't get a thread. One slow thing takes down the whole service. This is called cascading failure.

How to prevent it:

  • Timeouts — don't wait 5 seconds; give up quickly so the thread is freed. This is the most important fix.
  • Bulkhead — give gateway calls their own small pool, so if it fills up, the rest of the app keeps working.
  • Circuit breaker — if the gateway keeps failing, stop calling it for a short while and fail fast, giving it time to recover.
  • Retry carefully — only retry safe operations, with a small wait, not endlessly.
  • Fallback — return a polite "try again" instead of hanging.

You said "retry." On a payment. Be very careful. What's the danger?

ANSWER

Charging the customer twice. Maybe the payment actually went through, but the reply got lost, so we think it failed and retry — now they paid twice. The fix is an idempotency key: the request carries a unique id, and the payment system remembers it. If the same id comes again, it knows it's the same payment and returns the first result instead of charging again. Only retry payments when you have this in place.

A circuit breaker has three states. Name them and tell me what specifically moves it between each.

ANSWER
  • Closed — normal, calls go through. If too many calls fail, it switches to Open.
  • Open — stop calling, fail instantly for a short cooldown. After the cooldown, it switches to Half-Open.
  • Half-Open — let a few test calls through. If they work, go back to Closed. If they fail, go back to Open.

It's like checking if a sick service has recovered before sending it full traffic again.

4. Spring Boot & Spring Internals

"Everyone lists Spring Boot. Almost nobody understands the magic. Let's see."

CORE Singleton-scoped Spring bean is the default. You have a controller (singleton) with an instance field you write to per-request. What's wrong?
SIMPLE ANSWER

By default Spring makes only one copy of each bean (a "singleton") and shares it for all requests. If you store request data in a field on that shared object, two users hitting it at the same time will overwrite each other's data. So you get wrong, mixed-up results. Rule: keep beans without changing fields. Put per-request data in local variables (inside the method), not in shared fields.

But Spring lets me inject a prototype/request-scoped bean into my singleton. If I do, does it get a fresh one per request?

ANSWER

No, not automatically — this trips people up. A singleton gets its dependencies once when it's created, so it keeps using that same one. To truly get a fresh one each time, you need a scoped proxy (@Scope(value="request", proxyMode=TARGET_CLASS)) or inject a provider you call each time. The proxy quietly fetches the right per-request object on every call.

DEEP Explain @Transactional. Then tell me why calling a @Transactional method from another method in the SAME class does nothing.
SIMPLE ANSWER

@Transactional means "do all of this as one unit — if anything fails, undo everything." Spring does this by wrapping your bean in a proxy (a wrapper). When an outside caller calls your method, the wrapper starts the transaction, runs your code, then saves it (commit) or undoes it (rollback) if there's an error.

The same-class trap: if method a() calls b() in the same class directly, the call goes straight to b() and skips the wrapper. So b()'s @Transactional is ignored. Fix: move b() to a different bean, so the call goes through the wrapper.

By default, does @Transactional roll back on a checked exception? Trick question — be precise.

ANSWER

No. By default Spring only undoes (rolls back) on a runtime exception. If a checked exception (like IOException) is thrown, Spring actually saves the half-done work. Many people get burned by this. If you want it to roll back on any exception, write @Transactional(rollbackFor = Exception.class).

Difference between propagation REQUIRED and REQUIRES_NEW, and when does REQUIRES_NEW cause a self-deadlock?

ANSWER

REQUIRED (the default) means "join the transaction already running, or start one if none." REQUIRES_NEW means "always start a brand new separate transaction" — useful for things like saving a log that must stick even if the main work fails. The deadlock risk: the outer transaction locks a row, then the new inner transaction tries to lock the same row. The inner waits for the outer to finish, but the outer is waiting for the inner — they freeze each other.

5. Spring Security & OAuth 2.0

"You wrote 'OAuth 2.0' and 'security compliance.' I'm going to find out if you actually understand tokens or just configured a starter."

CORE You said you used OAuth 2.0. OAuth 2.0 is an authorization framework, not authentication. So what were you actually doing — and don't conflate it with OpenID Connect.
SIMPLE ANSWER

Good catch. Two different ideas:

  • Authentication = "who are you?" (login)
  • Authorization = "what are you allowed to do?" (permissions)

OAuth 2.0 is about authorization — it gives out a token that says what the holder is allowed to access. OpenID Connect is a small add-on on top that also handles "who are you" (login). In our system, when one service calls another with no human involved, we used the client credentials flow — the service gets a token to prove it's allowed.

Your token is a JWT. The gateway validates it. Does it call the auth server on every request to check it? Yes or no, and defend it.

ANSWER

No. A JWT is a signed token — like a sealed ID card. Our service just checks the signature using a public key it already has. No need to phone the auth server every time, so it's fast. (The other kind, an "opaque" token, would need a call every time.)

"Hard to revoke." So a stolen JWT is valid until it expires. How do you handle revocation and logout then?

ANSWER

True — a JWT stays valid until it expires, which is the downside. So we: (1) make tokens short-lived (a few minutes), so a stolen one is useless quickly; (2) keep a blocklist in Redis of tokens we want to kill early, and check it; (3) in a serious breach, change the signing key so all old tokens become invalid at once.

Where do you store the JWT on the client? If you say localStorage, I'm worried.

ANSWER

Not in localStorage — any malicious script on the page can read it (an XSS attack) and steal the token. For a website, store it in an HttpOnly cookie so JavaScript can't read it. For a mobile app (like MyJio), use the phone's secure storage (Keystore/Keychain).

DEEP Walk me through the Spring Security filter chain. Where does authentication happen and where does authorization happen?
SIMPLE ANSWER

Think of it like security checkpoints a request passes through, one after another:

  • First, an authentication filter checks who you are — it reads your token, verifies it, and remembers who you are for this request.
  • Later, an authorization filter checks if you're allowed to do this specific thing, based on your roles.
  • If login is missing or wrong, you get a 401. If you're logged in but not allowed, you get a 403.

So first "who are you," then "are you allowed."

You said the security context uses a ThreadLocal. Connect this to something you told me earlier about thread pools.

ANSWER

The "who you are" info is stored on the current thread. Since threads get reused, Spring clears it at the end of each request — otherwise the next request on that thread could think it's the previous user (a big security bug). Also, because it lives on the thread, if you start a new background thread yourself, that info does not carry over automatically; you have to pass it along.

6. Redis & Caching Strategy

"Redis is on your resume as the hero of your latency win. Let's make sure you didn't just call get and set."

CORE Name the caching patterns. Which did you use, and why not write-through?
SIMPLE ANSWER

Main ways to cache:

  • Cache-aside — app checks the cache first; if it's missing, read the database and put it in the cache. Most common. This is what I used.
  • Read-through — the cache itself loads from the database when missing.
  • Write-through — every write goes to cache and database together (keeps them in sync, but slows writes).
  • Write-behind — write to cache now, save to database later (fast, but you can lose data if it crashes).

I picked cache-aside because we read a lot and write rarely. Write-through would slow down writes for no real benefit, and write-behind could lose data — too risky near payments.

Define cache penetration, cache breakdown, and cache avalanche. They're different. I want all three and the fix for each.

ANSWER
ProblemIn simple wordsFix
PenetrationPeople keep asking for something that doesn't exist anywhere, so it never gets cached and always hits the database.Cache the "not found" answer too, or use a quick filter to reject impossible keys.
BreakdownOne very popular item expires, and thousands of requests hit the database at the same moment.Let only one request rebuild it; the rest wait briefly.
AvalancheMany items expire at the exact same time (or Redis goes down), flooding the database.Add random time to expiry; keep Redis backed up; protect the database.

Your Redis instance dies completely at peak. 1.5M transactions are flowing. What happens to your service in the next 10 seconds?

ANSWER

If we did nothing: every request now misses the cache and hits the database at once. The database was sized assuming Redis takes 90% of the load, so it gets overwhelmed, slows down, and the whole service can fall over. To avoid this: (1) treat Redis as a helper, not a must-have — wrap cache calls so if Redis is gone we just go to the database without hanging; (2) run Redis with a backup copy so it recovers in seconds; (3) keep a tiny cache inside the app for the hottest items; (4) protect the database by rejecting some load if needed. Always assume the cache can disappear.

DEEP Redis is single-threaded for command execution. With 1.5M transactions a day hitting it, isn't that a bottleneck? How is single-threaded fast?
SIMPLE ANSWER

Redis runs commands one at a time on a single thread — and that's actually why it's fast and simple: no waiting on locks, no fighting between threads. Everything is in memory (RAM), so each command takes microseconds; one thread can still do hundreds of thousands per second. The thing to avoid is a slow command (like listing all keys) — since there's one line, one slow command blocks everyone. So use safe commands and never KEYS * in production.

You said each command is atomic. So how do you do an atomic read-modify-write across multiple commands — like a rate limiter counter check-and-increment?

ANSWER

The clean way is a small Lua script. Redis runs the whole script as one step with nobody else jumping in. So you can read the counter, compare it to the limit, and increase it — all safely together. That's how a correct rate limiter is built on Redis.

7. Kafka & Event-Driven Architecture

"'Implemented Kafka-based event-driven flows.' Event-driven systems fail in subtle ways. I live in those failures."

CORE Why Kafka and not a regular message queue like RabbitMQ? And explain the role of partitions.
SIMPLE ANSWER

Think of Kafka as a long log book of events that it keeps for a while. Many different readers can read it, and you can re-read old events if needed. A normal queue like RabbitMQ usually deletes a message once it's read. We chose Kafka for high volume, for letting many services react to the same events, and for being able to replay history.

Partitions are how Kafka splits a topic into parts so many readers can work in parallel. Inside one partition, order is guaranteed. Across partitions, there's no fixed order. More partitions = more parallel work.

"Ordering is guaranteed within a partition." So how do you guarantee that all events for a single customer's recharge are processed in order?

ANSWER

Use the customer id as the key. Kafka always sends the same key to the same partition. So all of one customer's events land in one partition and are read in order. We don't need global order — we just need per-customer order, and this gives exactly that.

Acknowledge modes / delivery semantics. Tell me how to get exactly-once, and whether you actually had it.

ANSWER

Three levels: at-most-once (might lose messages), at-least-once (never lose, but might get duplicates), and exactly-once (no loss, no duplicates). Exactly-once is possible inside Kafka but very hard once you also write to a database and call an outside payment system. Honestly, we used at-least-once plus idempotent consumers — meaning if a duplicate event arrives, we recognize its id and skip it. That gives the same safe result, and it's the real-world answer.

Consumer lag is climbing. Walk me through diagnosing and fixing it without losing ordering.

ANSWER

Lag means readers are falling behind producers. First check: did events suddenly spike, or did readers slow down (maybe a slow database call or one bad "poison" message stuck)? Fixes: add more readers (but you can't have more readers than partitions), make each reader faster, or move the stuck bad message to a side "dead-letter" topic so the rest can flow. Order stays safe because one partition is always handled by just one reader.

HOPE-CRUSHER You publish a "recharge succeeded" event to Kafka AND write to your Oracle DB in the same flow. The DB commit succeeds but the Kafka publish fails. Now your DB says success but no event fired. How do you fix this — and don't say "use a transaction across both," because you can't.
SIMPLE ANSWER

This is the classic "two writes" problem — you can't make the database and Kafka succeed-or-fail together as one. The clean fix is the Outbox pattern:

  • In the same database save as your business data, also insert the event into a small outbox table. Since it's one database save, they always succeed or fail together — no mismatch.
  • A separate small worker reads new rows from the outbox table and sends them to Kafka, marking them as sent.
  • That worker keeps retrying until Kafka accepts it, so the event always gets out eventually. Consumers handle possible duplicates by id.

So instead of an impossible "both at once," you do one safe database save plus a reliable background sender.

CDC with Debezium reads the redo log. What's the downside versus a simple polling publisher?

ANSWER

CDC means a tool watches the database's change log and turns changes into events — fast, and it doesn't add query load. The downside: it's more moving parts to run and set up, and it ties you to that database's internals. A simple polling worker (just checking the outbox table every so often) is much easier to build but adds a little query load and delay. I'd start with polling and move to CDC only when scale demands it.

8. SQL / Oracle Tuning

"You profiled and tuned Oracle queries. Let's see if you understand the database or just ran a wizard."

CORE A composite index on (a, b, c). A query filters on b = ? AND c = ? but not a. Does the index help? Explain the leftmost-prefix rule.
SIMPLE ANSWER

Usually no. An index on (a, b, c) is sorted by a first, then b, then c — like a phone book sorted by last name, then first name. If you don't know the last name (a), you can't jump to the right place. This is the leftmost rule: the index works only if your search starts from the first column. Searching by b and c alone can't use it well. Tip: put the column you always filter on first.

So how do you order columns in a composite index? Give me the principle.

ANSWER

Put the columns you check with "equals" first, and any range check (like "date after X") last. A range check stops the index from using the columns after it. So for WHERE status = ? AND created > ?, make the index (status, created), not the other way. And if the index already contains every column the query needs, the database doesn't even touch the table — even faster.

DEEP Explain the difference between an optimistic and pessimistic lock, and which you'd use for decrementing a limited recharge offer's remaining quota under high concurrency.
SIMPLE ANSWER

Pessimistic lock = "I expect a fight, so I lock the row first." Others wait until I'm done. Safe, but slow if many people want it.

Optimistic lock = "I expect no fight, so I don't lock. I just check at save time whether someone changed it. If yes, I retry." Fast when conflicts are rare.

For a hot offer where everyone grabs the same count, fights are common, so plain optimistic retries waste effort. The clean way is one atomic update: UPDATE ... SET qty = qty - 1 WHERE id = ? AND qty > 0. The database handles it safely in one step, and the qty > 0 check stops us from going below zero (no overselling). For very high traffic, I'd keep the counter in Redis with an atomic decrease and sync to the database in the background.

You moved the counter to Redis. Now Redis and Oracle can disagree. What's your consistency story when you oversell because Redis said yes but the reconcile fails?

ANSWER

Honest trade-off. Redis becomes the fast "yes/no, is there stock" gate, and the database catches up a moment later. To not oversell: (1) the Redis decrease is atomic and never goes below zero, so the gate itself is correct; (2) the database update is saved to a durable log and retried if it fails, so we never silently lose a count; (3) if overselling is truly not allowed (real money/regulated), I keep the database as the boss with that qty > 0 update and accept it's a bit slower. Pick based on whether overselling is "annoying" or "illegal." For a recharge offer, a tiny safety buffer is fine.

CORE What are the SQL isolation levels and which anomaly does each prevent? Then: what's the default in Oracle, and what surprises people about it?
SIMPLE ANSWER

Isolation levels decide how much one transaction can see another's half-finished work. From loosest to strictest:

LevelIn simple words
Read UncommittedYou can see others' unsaved changes (risky). Rarely used.
Read CommittedYou only see saved data, but the same query twice can give different results.
Repeatable ReadThe same query gives the same rows during your transaction.
SerializableStrictest — acts as if transactions ran one by one.

Oracle's default is Read Committed. The nice surprise: in Oracle, readers never block writers and writers never block readers — readers see a stable snapshot. So Oracle doesn't get stuck the way people expect from other databases.

9. Payments, Idempotency & Consistency

"You build payment APIs. This is where I have zero tolerance for hand-waving. Money bugs end careers."

DEEP Design idempotency for a "create payment" endpoint. The client times out and retries. Walk me through the exact mechanism so the customer is never charged twice — including the race where two identical requests arrive simultaneously.
SIMPLE ANSWER

The app sends a unique idempotency key (one id per payment attempt, reused if it retries). On the server:

  1. Try to save that key in a table with status "in progress." The key has a unique constraint, so the database allows only one.
  2. If the save works, this is the first time — do the payment, then store the result as "done."
  3. If the save fails (key already there), it's a repeat — if it's "done," return the saved result (no second charge); if it's still "in progress," tell the client to wait.

For the two-at-once race: both try to save the same key, but the database lets only one win. The other sees "already exists" and waits. So the database itself stops the double charge — no extra locking needed.

What's the TTL on that idempotency key, and what breaks if you pick it wrong in either direction?

ANSWER

Keep the key long enough to cover any retry — usually a day or more for payments. Too short: it expires, a late retry looks brand new, and you charge twice — the exact bug you were preventing. Too long / forever: storage keeps growing. So set it a bit longer than the longest possible retry, then let it expire.

The customer's money left their account but our DB write failed after the gateway charged them. Now there's money in limbo. How do you reconcile?

ANSWER

Payments must be built around checking and matching, not just one request/reply. So: (1) write a "pending" record before calling the gateway, so we always have a trail; (2) listen to the gateway's callback/webhook, which is the real source of truth, and update our records from it; (3) run a daily reconciliation job that compares our records with the gateway's report and fixes mismatches (charged but not recorded → record it or refund; recorded but not charged → mark failed). The customer is always made right by this matching loop.

HOPE-CRUSHER Explain the Saga pattern, then design a recharge saga (debit wallet → call telecom operator → confirm) and tell me exactly what happens if the operator call succeeds but the confirmation step fails.
SIMPLE ANSWER

A Saga is how you handle a multi-step process across services when you can't lock everything together. You do each step on its own, and for each step you also have an undo step in case a later step fails.

Recharge saga:

  1. Take money from wallet. Undo = refund the wallet.
  2. Tell the telecom operator to recharge. Undo = reverse with operator (if possible).
  3. Confirm and mark the order done.

If step 2 works but step 3 fails: the recharge already happened, so we must NOT refund (that gives a free recharge). Instead we retry the confirm step until it succeeds. Key idea: undo backward only for steps you can undo; for steps you can't undo (operator already recharged), go forward and finish the rest.

"Refund the wallet as compensation." But what if the compensation itself fails?

ANSWER

The undo steps must also be safe to retry, and we keep retrying them (from a durable queue) until they work. If one still can't succeed after many tries, it goes to a "needs a human" queue and raises an alert — because now real money is stuck and someone in finance must fix it. We also save the saga's progress, so if the system restarts, it knows where it stopped and continues.

10. Distributed Systems & System Design

"You wrote 'distributed systems' and 'system design best practices.' Prove you can think at the system level, not just the class level."

CORE State the CAP theorem precisely. Your recharge system — is it CP or AP, and why? Don't give me the textbook answer; tie it to a real decision.
SIMPLE ANSWER

CAP says: when the network between servers breaks (a "partition"), you must pick one — stay correct (Consistency) or stay available (Availability). You can't have both during that break.

My real choice: the money part picks correctness — if servers can't agree, I'd rather reject the payment than risk a wrong balance. A "try again" is fixable; a wrong balance is not. The browsing part (showing plans) picks availability — keep working even if the list is a few seconds old, because slightly old plans are harmless and we re-check the price at checkout. So different parts make different choices.

You said "eventual consistency" for the catalog. Define it without using the word "eventually," and tell me what a user actually experiences.

ANSWER

It means: after a change, all copies will match up within a short time, but for those few seconds different users might see slightly different versions. In practice a user might briefly see a plan that was just removed, or not yet see a brand-new plan. The promise is that everyone catches up soon — not instantly. Fine for a plan list, not fine for a bank balance.

DEEP Design a rate limiter for the API gateway — you claimed you built one. I want the algorithm, where the state lives, and why your choice handles a distributed multi-instance gateway correctly.
SIMPLE ANSWER

A rate limiter caps how many requests someone can make. Common methods:

  • Fixed window — count per minute. Simple, but allows a burst at the edge of two minutes.
  • Sliding window — smoother counting, more accurate.
  • Token bucket — you get tokens that refill steadily; each request spends one. Allows short bursts but caps the steady rate. My favorite for a gateway.

Why store it in Redis: we have many gateway servers. If each counted on its own, a user could hit each one and get many times the limit. So all servers share one counter in Redis, and we update it with a small Lua script so the check-and-count is one safe step. Over the limit → return 429 Too Many Requests.

Redis adds a network hop to every request at the gateway. At 1.5M+/day with peaks, isn't that latency and a SPOF? Optimize it.

ANSWER

Fair. Options: (1) let each server count locally and sync with Redis every so often — slightly less exact, but rate limiting doesn't need to be perfect, and it's much faster; (2) give each server a share of the total budget; (3) run Redis with a backup so it's not a single point of failure, and the call is sub-millisecond anyway; (4) if Redis is down, allow the traffic through (fail open) for non-critical endpoints rather than blocking real users. The idea: rate limiting can be approximate, so push the work close to each server.

HOPE-CRUSHER Two of your microservices need the same data. Do you share a database between them? Defend whatever you say — there's a wrong answer here.
SIMPLE ANSWER

No. Each service should own its own database. If two services share one database, a change by one team can break the other, and you can no longer deploy or scale them separately. That quietly turns your microservices back into a tangled monolith.

How they share data instead: (1) one service asks the other through an API; or (2) one service sends events and the other keeps its own small copy of just the fields it needs. I prefer events so a service isn't stuck waiting on another being up.

"Each service keeps a local copy." Now you have the same data in 3 places. How do you handle the inevitable drift and a "source of truth" question?

ANSWER

Pick one owner for each piece of data — only it can change the data and send updates. The others keep read-only copies. To keep copies correct: process events in order per item (key by id), include a version number so old/late updates are ignored, and run a periodic full re-sync to fix anything missed. If there's ever a disagreement, the owner wins, and anything that must be exact (like a balance) is read from the owner, not the copy.

DEEP How would you design MyJio's recharge to handle a 10× traffic spike during a festival promotion at midnight? Walk the whole path.
SIMPLE ANSWER
  • Cache early — warm up the caches before midnight so the first rush doesn't all hit the database.
  • Scale up before the spike — add more servers ahead of time (we know it's coming), not after it starts.
  • Use a queue — accept the request fast, put it on Kafka, and process it just behind. This soaks up the spike instead of crushing everything. The user sees "processing," then a confirmation.
  • Protect the database — it's the hardest to scale. Cache hard, use read copies, and limit how fast it's hit.
  • Stay resilient — circuit breakers and timeouts around the payment gateway and operator, and drop non-essential work if needed.
  • Idempotency everywhere — retries jump during a spike, so make repeats safe.
  • Watch live — dashboards and alerts during the event.

Main idea: turn a sudden rush into a smooth queue, add servers ahead of time, and shield the database.

11. Kubernetes / OpenShift

"You deploy on K8s/OpenShift with 'concurrency tuning.' Most devs treat it as a black box. I won't let you."

CORE Difference between liveness, readiness, and startup probes. What happens if you misconfigure liveness to point at a slow dependency?
SIMPLE ANSWER

These are health checks Kubernetes runs on your app:

  • Liveness — "is the app stuck/dead?" If it fails, Kubernetes restarts it.
  • Readiness — "can it take traffic right now?" If it fails, Kubernetes stops sending traffic but does not restart it.
  • Startup — "has it finished booting?" Gives a slow-starting app time before the other checks begin.

The mistake: if liveness checks a dependency like the database, and the database has a brief hiccup, then every pod fails liveness and Kubernetes restarts them all at once — turning a small hiccup into a full outage. And restarting doesn't even help, because the database was the problem. Rule: liveness should only check the app itself; check dependencies in readiness.

DEEP Your Java pod keeps getting OOMKilled even though the JVM heap looks fine. Why, and how do you fix it?
SIMPLE ANSWER

"OOMKilled" means the container used more memory than its limit, so Kubernetes killed it. The trap: Java uses more memory than just the heap. There's also thread stacks, class metadata, network buffers, and more. So the heap can look healthy while the total memory goes over the container limit.

Fixes: (1) don't set the heap to the full container size — leave room for the extra stuff (use a percentage like 75%); (2) limit the other memory areas and watch the thread count; (3) size the container = heap + extras + a little buffer; (4) if memory keeps growing, it's a real leak — take a heap dump and find what's holding on.

Before container-awareness, what was the classic bug with the JVM on K8s?

ANSWER

Old Java versions didn't understand container limits. A pod limited to 1GB on a big 64GB machine would think it had all 64GB and all the CPUs, so it sized its memory and threads way too big — then got killed instantly. Newer Java reads the container limits correctly, which is why keeping an up-to-date Java version matters.

CORE During a rolling deployment, in-flight requests get dropped and users see errors. Walk me through making it zero-downtime.
SIMPLE ANSWER

The errors happen because Kubernetes shuts down an old pod while it's still handling requests. To avoid it:

  • Graceful shutdown — when told to stop, the app stops taking new requests, finishes the ones in progress, then exits. Spring Boot supports this.
  • Small wait before stopping — give Kubernetes a moment to stop routing traffic to the pod before it shuts down, so no new request arrives mid-shutdown.
  • Start new before killing old — bring up new pods and make sure they're ready before removing the old ones.
  • Retries — let the client retry safe requests to cover any tiny gap.

12. SOLID & Design Patterns

"You wrote 'applying SOLID principles.' Anyone can recite the acronym. I want a real example where you applied one — and where you'd violate one on purpose."

CORE Recite SOLID, then give me a concrete refactor from your work where the Dependency Inversion Principle mattered.
SIMPLE ANSWER

SOLID is five rules for clean code:

  • S — one class, one job.
  • O — add new features by adding code, not by editing old code.
  • L — a subclass should work anywhere its parent works, without surprises.
  • I — many small interfaces are better than one giant one.
  • D — depend on an interface, not a specific class.

Real example (the D): our recharge code used to call one specific payment gateway class directly. When we added a second gateway, I made a PaymentGateway interface, and the code now talks to the interface. Adding a new gateway is just a new class — no change to the main logic. Bonus: it's easy to test with a fake gateway.

Give me a Liskov violation that compiles fine but is a real bug. The classic one.

ANSWER

The Square extends Rectangle example. A square forces width and height to be equal. So code that sets width = 5 and height = 4 and expects area = 20 breaks when given a square (it gets 16). The code compiles, but the square doesn't behave like a normal rectangle — that's a Liskov violation. The fix is to not force that broken inheritance.

When would you deliberately violate SOLID? If you say "never," you're not senior.

ANSWER

SOLID is a guide, not a law. I break it when following it adds more complexity than it saves. For example, I won't add an interface "just in case" if there's only one implementation — that's extra clutter for no reason (add it when a second case actually shows up). A small simple data object doesn't need to be split into five tiny classes. The senior move is knowing SOLID exists to make change easier — if a piece won't change, over-engineering it is its own mistake.

13. Projects & Research

"Let's talk about what you built outside Jio — and your publications. I'll test whether you understand them or just have your name on them."

WARMUP Your Job Portal uses Spring Security with role-based dashboards (Admin, Employer, Candidate). How exactly did you enforce that an Employer can't hit an Admin endpoint?
SIMPLE ANSWER

Two layers. First, in the security config I map URLs to roles — only ADMIN can reach /admin/**, only EMPLOYER can reach /employer/**. Second, I add a role check on the important methods too (@PreAuthorize), so even if a URL rule is wrong, the method still blocks it. The roles come from the logged-in user. Importantly, I never rely on just hiding buttons in the UI — the server always enforces it, because the client can be faked.

An Employer changes the id in the URL to view another employer's job applications. Your role check passes — they ARE an employer. What's the vulnerability and the fix?

ANSWER

This is IDOR — checking the user's role but not whether they own that specific record. They're a valid employer, so the role check passes, but the record isn't theirs. The fix: always check ownership too — the query should be "this job AND owner = current user." Never trust an id from the URL as proof the user is allowed to see it.

DEEP Your CNN+LSTM depression-detection paper. Why both CNN and LSTM? Why not one? And what's the danger of your reported accuracy?
SIMPLE ANSWER

They do different jobs. A CNN is good at spotting small local patterns — like key phrases in the text. An LSTM is good at understanding a sequence in order — how words build meaning across a sentence. Stacking them lets the CNN pick out useful pieces and the LSTM understand how they flow. It was a strong combo for text before transformers became common.

The danger of accuracy: if most samples are "not depressed," a model can score high just by always guessing "not depressed" — accuracy looks great but it's useless. For a health topic, the number that matters is recall (did we catch the truly at-risk people?), plus precision and F1. I wouldn't claim it's medically reliable — it was a research project.

Today, in 2026, would you still use CNN+LSTM for this? If not, what, and what's the catch?

ANSWER

No — today I'd use a transformer (like a fine-tuned BERT model). It understands long text better and trains faster in parallel. The catch: transformers need lots of data and compute, and can be overkill for a small dataset. And for a sensitive mental-health use, the bigger concerns become fairness, privacy, and being able to explain the decision — not just the score.

14. Rapid-Fire Hope-Crushers

"We're near the end. Short questions, no time to think. This is where I find out what's actually in your head versus what you rehearsed."

RAPID Difference between PUT and PATCH, and why does it matter for idempotency?
ANSWER

PUT replaces the whole record — doing it many times leaves the same result (idempotent, safe to retry). PATCH changes part of it and may not be safe to repeat (e.g., "add 10" runs again and again). So on a timeout, PUT is safe to retry; PATCH needs an idempotency key. (Also: GET, PUT, DELETE are idempotent; POST is not.)

RAPID What does @Transactional(readOnly = true) actually buy you?
ANSWER

It tells the system "this only reads, never writes." Then Hibernate skips checking for changes (less work), and the database can use a lighter mode or a read copy. It makes read operations a bit faster and prevents accidental writes.

RAPID N+1 query problem in JPA/Hibernate — what is it and how do you kill it?
ANSWER

You load a list of N things, then for each one it secretly runs one more query to load its related data — so 1 + N queries, which is slow. Fix: load them together in one query with JOIN FETCH (or an @EntityGraph). The trap is it hides until you look at the SQL log, so always check the queries being run.

RAPID 502 vs 503 vs 504 — what's actually broken in each?
ANSWER

502 — the gateway got a bad reply from the server behind it. 503 — the server is up but too busy or in maintenance ("not now"). 504 — the gateway waited for the server and got no reply in time (too slow). Quick memory: 502 = bad reply, 503 = not now, 504 = too slow.

RAPID Your connection pool is exhausted. HikariCP. What three numbers do you check first?
ANSWER

(1) Active vs max connections — are they all in use? (2) How long requests wait to get a connection. (3) How long each connection is held — usually the real cause is connections held too long (a slow query, or a transaction kept open across a network call). Adding more connections rarely fixes it; it just pushes the problem to the database.
