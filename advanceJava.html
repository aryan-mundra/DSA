50. Difference between Future and CompletableFuture?

Future and CompletableFuture, both help with asynchronous programming, but they’re quite different in how they work.

Real time example: Imagine you are ordering a pizza.

Scenario A: You went to the restaurant and ordered and wait there on the table. You are not doing anything else. Just sitting and waiting for your order. That is Future.

Scenario B: You place the order through an app from home. While it is being prepared, you watch Netflix, take a shower, and the app notifies us when it is done. That is CompletableFuture.

51. How Java handles exceptions?

An exception in Java is just an unexpected situation that breaks the normal flow of the program. Java handles exceptions using a structured mechanism that involves keywords: try, catch, finally.

  • try block: This block encloses the code segment that is susceptible to throwing an exception.
  • catch block: If an exception occurs within the try block, the corresponding catch block is executed. A try block can have multiple catch blocks to handle different exception types.
  • finally block: This block is optional and contains code that is guaranteed to execute, regardless of whether an exception occurred in the try block or was caught by a catch block. It is typically used for cleanup operations like closing resources. 
try {
  BufferedReader reader = new BufferedReader(new FileReader("data.txt"));
  String line = reader.readLine();
  System.out.println(line);
} catch (IOException e) {
  System.out.println("Something went wrong while reading the file.");
  e.printStackTrace();
} finally {
  System.out.println("Cleanup code goes here.");
}

53. What is the difference between throw and throws?

Throw:

The throw keyword is used to manually throw an exception in the code. When throw is executed, the normal flow of execution is immediately terminated, and the control jumps to the nearest catch block that can handle the thrown exception. If no such catch block is found, the exception propagates up the call stack.

Real-World Example: Imagine we are writing a banking app. If a user tries to withdraw more money than they have, we throw an exception.

public class BankAccount {
  private double balance;

  public void withdraw(double amount) {
    if (amount > balance) {
      throw new IllegalArgumentException("Insufficient funds!");
    }
    balance -= amount;
  }
}

Throws:

throws is used in a method's signature to declare the types of checked exceptions that the method might throw. This informs calling methods that they need to either handle these exceptions (using try-catch) or declare them as well (propagating them further).

Real World Example: We are writing a file reader method. Since files can be missing or corrupted, Java forces us to declare throws IOException to warn callers. 

public class FileProcessor {
  // Warns that this method might throw an IOException
  public String readFirstLine(String filePath) throws IOException {
    BufferedReader reader = new BufferedReader(new FileReader(filePath));
    return reader.readLine();
  }
}

Yes, but we can throw narrower checked exceptions. We can’t add new broader checked exceptions in the child class. 

class Parent {
  void read() throws java.io.IOException {}
}

class ChildOk extends Parent {
  // narrower checked exception is allowed (subclass of IOException)
  @Override
  void read() throws java.io.FileNotFoundException {}
}

class ChildNotOk extends Parent {
  // NOT allowed: broader/new checked exception than parent declares
  @Override
  void read() throws java.lang.Exception {}
}

54.1 How to create it?

We create custom exceptions by extending either Exception (checked) or RuntimeException (unchecked).

Real-world: For an e-commerce stock issue, we might create OutOfStockException. 

// Checked custom exception (caller must handle or declare)
class OutOfStockException extends Exception {
  OutOfStockException(String message) {
    super(message);
  }
}

// Unchecked custom exception (optional to handle/declare)
class OutOfStockRuntimeException extends RuntimeException {
  OutOfStockRuntimeException(String message) {
    super(message);
  }
}

54.2 When to extend Exception vs RuntimeException?

  • Extend Exception (checked): If the exceptional condition is recoverable or expected to be handled by the caller.
  • Extend RuntimeException (unchecked): If the exceptional condition is unexpected or typically indicates a programming error.

Real-world examples:

  • Checked (Exception): A user tries to buy 5 items but only 2 are in stock. The caller (service/controller) can show a proper message and let the user change quantity.
  • Unchecked (RuntimeException): A bug causes stock to be negative due to wrong business logic, or a required value is unexpectedly null—this is more like a programming error that should be fixed.

55. Explain equals() and hashCode() contract?

equals() tells if two objects should be treated as the same logically. By default (from java.lang.Object), it compares memory address (same object in RAM). If you want logical comparison (like comparing two users by id), you override equals().

hashCode() gives a number that helps hash-based collections (like HashMap, HashSet) quickly find the object.

Real-world example:

Think of a delivery address. hashCode() is like a pincode / area code that quickly groups addresses into the same area (fast to narrow down). equals() is like the full address (house no + building + street) that confirms it’s exactly the same place (final confirmation).

Contract:

  • If two objects are equal (equals() returns true), they must have the same hashCode() value.
  • If two objects have the same hashCode(), they might still not be equal.

When different objects end up with the same hashCode(), it’s called a hash collision. Then Java uses equals() to check the real match.

57.1 Can I insert in the middle of ArrayList?

Yes. You can insert in the middle using add(index, element). But it takes O(n) time because elements after that index must shift right.

Example in Java:

ArrayList<Integer> list = new ArrayList<>();
list.add(10);
list.add(20);
list.add(30);

list.add(1, 15); // inserts in the middle (between 10 and 20)

System.out.println(list); // [10, 15, 20, 30]

57.2 How does Java manage dynamic resizing in ArrayList?

When the internal array is full, ArrayList creates a new array with about 50% more capacity, then copies all existing elements into the new array.

This copy step is why resizing is expensive sometimes.

57.3 Why is LinkedList not preferred for random access?

LinkedList stores elements as nodes with references to the next and previous nodes. To access element at index i, it must traverse from the head or tail until it reaches that node, which is O(n).

ArrayList can do get(i) in O(1) because it uses array indexing.

Back to top

57.4 If you are inserting an element in the middle of a LinkedList with 10 million items, how will it perform compared to an ArrayList?

LinkedList: To insert in the middle, you first traverse to that position (O(n)), then update pointers (O(1)). Overall: O(n).

ArrayList: Inserting in the middle requires shifting all elements after that index (O(n)).

For 10 million items: both are O(n), but in practice LinkedList can be slightly faster for the insertion step because there is no shifting—only pointer updates (after you reach the node).

Rule:

  • Use LinkedList if frequent insertions/deletions in the middle are needed.
  • Use ArrayList if random access and iteration speed matter more.
Back to top

57.5 If you need to implement a queue that frequently adds and removes elements from both ends, which list would be a better choice and why?

Pick LinkedList over ArrayList. LinkedList implements Deque, so it is designed for efficient insertions and deletions at both ends in O(1) time.

ArrayList is backed by an array, so adding/removing at the front (or middle) requires shifting elements, which is O(n).

57.6 Defining the constructor capacity in ArrayList increases performance?

Yes, it can. If we don’t set capacity, ArrayList starts with a small default size and grows by creating a bigger array (usually ~1.5x), copying elements, then continuing. This resizing and copying costs time and memory.

So if you already know you will store, say, 10,000 items, setting an initial capacity can save multiple resizes. 

List<Integer> list = new ArrayList<>(10000);

59.5 Building a cache: fast access and predictable iteration order

You're building a cache: you store key–value pairs and look them up often (fast access).

You also care how entries appear when you loop over the map—for example oldest-first, insertion order, or “who was used least recently.” That's predictable iteration order (not random like a plain HashMap).

So the question is: which Map gives you both (1) quick get/put and (2) a stable, meaningful order when you iterate?

LinkedHashMap is chosen because it's basically HashMap + a linked list that remembers order:

“Fast access like a HashMap”

Lookups still use hashing, so average-time behavior is like a normal hash map (fast for typical cache use).

“Maintains order”

When you iterate entrySet() / keySet(), entries follow insertion order (unless you switch it to access order, which is a LinkedHashMap option—useful for LRU-style behavior).

“LRU caching”

LRU = “throw out the least recently used item when the cache is full.”

Many LRU cache examples use LinkedHashMap because you can:

  • iterate in a fixed order (who to evict next), and
  • still do fast get/put.

59.7 What are the consequences of non-final fields in a class that is used as a key in a HashMap?

Using non-final fields in a class that is used as a key in a HashMap can cause serious issues because HashMap depends on two things:

1. 2. Problem:

hashCode(): determines which bucket the key goes into.

equals(): determines whether two keys are the same inside that bucket.

If the fields used in hashCode() or equals() are mutable (non-final), and if we change them after inserting the object into the map, then:

  • The key's hashCode() may change, so the object is now in the wrong bucket.
  • A lookup using the same key (but with changed values) will fail , we will get null even though the entry exists. 
class Person {
  String name; // mutable, not final

  Person(String name) {
    this.name = name;
  }

  @Override
  public int hashCode() {
    return Objects.hash(name);
  }

  @Override
  public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Person)) return false;
    Person p = (Person) o;
    return Objects.equals(name, p.name);
  }
}

public class HashMapProblem {
  public static void main(String[] args) {
    Map<Person, String> map = new HashMap<>();
    Person p1 = new Person("Raju");
    map.put(p1, "Engineer");
    // Lookup works
    System.out.println(map.get(p1)); // Engineer
    // change the key
    p1.name = "Raghu";
    // Lookup fails
    System.out.println(map.get(p1)); // null
  }
}
  • When we put p1 in the map, its hash is based on "Raju".
  • After changing p1.name to "Raghu", its hash changes but the entry is still stored under the old bucket.
  • So, lookup fails, the object exists in the map, but HashMap can't find it anymore.
Back to top

59.8 What happens if we have added mutable object to HashMap and change it.

It depends whether that object is the key or only the value.

If the object is the key and you change fields that are used in hashCode() or equals(), the map can “lose” the entry: get may return null even though the pair is still inside the map, because the map still looks in the old bucket from the old hash. (Same idea as 59.7.)

If the object is only the value, changing it does not move the entry: you can still find it with the same key; only the stored object’s contents change.

61.1 What is fail-fast and fail-safe?

Fail-fast (simple)

Imagine a teacher is reading names from the same attendance sheet you are still allowed to edit. If a line is added or removed while the teacher is reading, the teacher stops immediately and says “this list changed while I was reading it.” In Java, that kind of stop is usually a ConcurrentModificationException. The goal is to avoid silently showing wrong or mixed-up data.

What it means in code words

A fail-fast iterator notices when the collection was structurally changed (add/remove, etc.) after the iterator was created, while you are still iterating. It usually throws ConcurrentModificationException right away instead of letting you keep going with inconsistent results.

Typical example

Iterating a HashMap with an Iterator / enhanced-for and calling map.put(...) inside the loop.

Fail-safe (common interview meaning) (simple)

Imagine the teacher reads from a photocopy of the list. Students may change the real list on the desk, but the photocopy in the teacher’s hand does not change during that read, so the teacher usually does not crash. You might not see the newest changes on that one pass—that is the trade-off.

What it means in code words

A fail-safe iterator is designed so that concurrent changes to the underlying collection do not make the iterator throw (at least not for the same reason as fail-fast). Often it does this by iterating over a snapshot or a weakly consistent view (depending on the class).

Typical examples

Iterators on ConcurrentHashMap, CopyOnWriteArrayList (iterates a copy), etc.

61.8 Suppose I am using a Hashtable, one thread is updating the Hashtable and another thread at the same time is getting the value—then what happens?

Hashtable synchronizes its public methods on the table itself. So put / remove / get (and similar) do not truly run at the same time on the same instance: one call holds the lock, the other waits, then runs on a consistent internal state.

In your scenario, if one thread is updating while another calls get, the usual outcome is that the get either happens before or after the update (depending on which thread gets the lock first)—not both mutating/reading the internal structure concurrently without ordering.

Note: safety is still per method. Multi-step logic like “check then put” needs extra synchronization or a higher-level concurrent map.

61.9 Suppose I am using a ConcurrentHashMap, one thread is updating it and another thread at the same time is getting a value—then what happens?

ConcurrentHashMap is built for this: one thread can put while another gets, and the map stays safe (it won’t “break inside” like a normal HashMap race).

The reader usually sees either the old value or the new value for that key, depending on tiny timing—even if both threads feel “at the same instant,” the CPU still applies an order under the hood.

62. ConcurrentHashMap vs Synchronized HashMap?

Here “Synchronized HashMap” means the thread-safe map from Collections.synchronizedMap(new HashMap<>()).

# Topic ConcurrentHashMap Synchronized HashMap
1 Locking mechanism

The whole map is not locked. The map is divided into a number of segments, and each segment maintains its own lock.

In simple words: Imagine a big office split into several rooms. If two people need to change files in different rooms, they can often work at the same time, because each room has its own lock. They only wait if two people need the same room at once.

That is the idea of fine-grained locking: you don’t lock the entire building for every small change—only the part that truly needs protection.

Maintains an object-level lock—i.e. the whole map is locked.

In simple words: Imagine one main door for the whole office. Before any work on the map (get, put, etc.), a thread must go through that same door. While one thread is inside, others typically wait outside —even if they only wanted a quick read.

So it is easy to understand and safe for simple uses, but it can become a bottleneck when many threads hit the map often.

2 Synchronized operations

Only update operations are synchronized. Read operations are not synchronized (in the same “whole-map lock” sense).

If Thread A updates while Thread B calls get on the same key—what value does B see? Usually either the old value (before the update is fully visible) or the new value (after the update is visible). Think “before photo vs after photo,” not a random broken value.

Why the table talks about locks: ConcurrentHashMap is built so put/remove still happen safely while many threads read. A synchronized HashMap wrapper is simpler: even a get often waits in the same single lock line as writers, so readers and writers block each other more.

One caution (different issue): “Check map, then put if missing” is two steps. Another thread can change the map between those steps—use one atomic helper like putIfAbsent (or similar) if you need that guarantee.

All operations are synchronized—whether read or update, the thread must acquire the object lock.

In simple words: Every common map method is wrapped with synchronized on the same monitor. So a get must wait behind a put (and vice versa) if they happen together—because they all share the one big lock strategy.

That makes behavior predictable for beginners (“everything is guarded”), but under heavy traffic threads spend more time waiting in line, which hurts throughput.

63. Why ConcurrentHashMap won't allow null keys and null values?

In a normal HashMap, we can store one null key and many null values. That is fine because everything happens in a single-threaded context.

But in a ConcurrentHashMap, multiple threads read and write at the same time. If null were allowed, two big problems show up:

The problem with null keys

Null key in HashMap (single-threaded)

In a normal HashMap, if we put null as a key, Java keeps it in a special way (often described as bucket 0 / a dedicated path). Since only one thread is using the map in a typical single-threaded context, that is fine.

Problem in ConcurrentHashMap

ConcurrentHashMap is different:

  • It splits data into many buckets.
  • Keys go into buckets based on their hashCode().
  • This allows many threads to work on different buckets at the same time.

But with null as a key:

  • There’s no hashCode() to calculate.
  • If all null keys were forced into one bucket (for example bucket 0), then all threads must fight over that one bucket.
  • This slows things down and makes concurrency messy.

The problem with null values

Imagine a scenario where null values are allowed. If we call map.get(key) and it returns null, what does that mean? In a single-threaded HashMap, null could mean either:

  1. The key doesn’t exist in the map.
  2. The key exists, but its value is explicitly set to null.

In a concurrent setting, this ambiguity becomes a nightmare:

  • Thread A checks map.get(key) and gets null, assuming the key doesn’t exist.
  • But Thread B might have just set the value to null using map.put(key, null).

Now Thread A can’t tell whether the key is absent or its value is null, leading to race conditions or incorrect logic. By banning null values, ConcurrentHashMap ensures that get() returning null always means the key doesn’t exist.

64.9 What happens if you add elements to a HashSet with duplicate hashCode() values but different equals() implementations?

If two objects have the same hashCode() but equals() returns false, the HashSet will store both objects.

Here is why:

  • HashSet first checks the bucket using hashCode().
  • Within the bucket, it uses equals() to see if the object already exists.
  • Since equals() is false, HashSet treats them as different, even though they collide in the hash bucket.

In depth (where they are stored):

A HashSet is backed by a HashMap. The element you add becomes the key in that internal map. The map first maps the object’s hashCode() to a bucket index (a “bin”). Many different objects can map to the same bucket—that is normal and is called a hash collision.

Inside that bucket, Java does not store “only one object.” It stores a chain (or similar structure) of entries that all landed in that bucket. When you add the second colliding object, the map walks that chain and compares with equals(). Because equals() is false, it decides “this is a different element,” so it adds another entry in the same bucket.

So both objects live in the same bucket, but as separate entries linked together—same hashCode() address, different equals() identity.

65. Explain about Queue?

A Queue in Java is a linear data structure that follows the First-In-First-Out (FIFO) principle, meaning the first element added to the queue will be the first one to be removed. Queues are widely used in scenarios like task scheduling, buffering and producer consumer problems.

Since the Queue is an interface, we cannot provide the direct implementation of it.

Queues in Java can be classified into bounded and unbounded implementations based on their capacity.

Bounded vs unbounded (idea)

  • Unbounded queue: Can grow dynamically (no fixed size).
  • Bounded queue: Has a fixed maximum capacity. A bounded queue has a fixed size. Once it is full, we can’t just keep adding elements—either the operation fails, or it waits until space is available.

Bounded queue examples (common interview classes)

  • ArrayBlockingQueue: It uses an array with a fixed capacity. It also supports fair ordering if we want threads to access in a strict first-come, first-served manner.
  • LinkedBlockingQueue: Works like a linked list and we can set the capacity. If we don’t, it defaults to a very large size (Integer.MAX_VALUE).
  • PriorityBlockingQueue: Instead of strict FIFO, it orders elements based on priority (either natural ordering or a custom comparator).

Unbounded queues

Unbounded queues don’t force a hard capacity limit. They’ll grow as needed, limited only by the available memory.

  • PriorityQueue: Keeps elements ordered by natural order or a comparator, but it is not thread-safe.
  • ConcurrentLinkedQueue: A thread-safe option that works well in highly concurrent scenarios.
  • LinkedList (as Queue): LinkedList also implements Deque. So, it can also be used as Queue.

66. When will you use each queue?

Below is a practical “when to use it” line for each type, plus one real-world use case you can say in interviews.

ArrayBlockingQueue

When: You need a fixed maximum capacity FIFO queue, often shared by multiple threads, and you want predictable memory (array-backed).

LinkedBlockingQueue

When: You want a classic producer–consumer queue; optionally bounded, and linked-node structure is fine.

PriorityBlockingQueue

When: Multiple threads produce/consume work, but the “next item” must be chosen by priority (not strict FIFO).

When: Many threads enqueue/dequeue frequently and you want a thread-safe, lock-free style FIFO queue for high concurrency.

LinkedList (as Queue)

67. How ArrayList works internally?

Answer (how it really works)

ArrayList is not a special magic list—it is mostly a thin wrapper around a normal Java array (conceptually Object[]) that stores references to your elements in contiguous indexes 0 .. capacity-1.

Two numbers matter all the time: capacity (how many slots exist in the backing array) and size (how many slots are actually used from index 0 to size-1). When you call add(x) at the end, ArrayList writes into index size (if space exists) and then increases size by 1. That end-add is why it is usually fast.

When size would become larger than the current capacity, ArrayList must grow. Growing means: allocate a new bigger array (often about 1.5× the old capacity in modern JDKs), copy all elements from the old array to the new array, then start using the new array. That copy is why an occasional add can take noticeably longer.

get(i) is fast (O(1)) because an array supports direct indexing: the CPU/JVM can jump straight to slot i. Inserting/removing in the middle is slower (O(n)) because indexes must stay “dense”: inserting at i pushes elements from i onward one step to the right; removing pulls them left.

68. How HashMap works internally?

HashMap contains an array of nodes, and each node is represented as a class (a Node<K, V>-style record). It uses that array plus linked lists (chains of next pointers) internally for storing keys and values. In the classic picture there are four fields per node:

  • int hash — derived from the key’s hash code (used for bucket choice and tuning)
  • K key
  • V value
  • Node<K, V> next — points to the next entry in the same bucket when hashes collide

At the beginning, Node<K, V>[] table is allocated (for example default length 16, so indices 0–15), and every bucket is null until the first put. So HashMap is “array of heads of small linked structures,” not one giant list.

On put(key, value), the map hashes the key, maps that to a bucket index, and either places a new node at that head or walks the chain using equals to find an existing key (to overwrite the value) or to append at the end of the chain. On get(key), the same bucket index is computed, then the same chain walk finds the matching key or returns nothing.

hashCode() narrows the search to one bucket; equals() confirms the exact key among nodes in that bucket. Equal keys must have consistent hashes, or the map breaks. When too many entries live in the table relative to capacity, the array resizes (larger table), entries are rehashed into new indices, and chains tend to shorten again. The default load factor (0.75) controls when that resize happens.

In modern Java, if a single bucket’s chain grows very long, that bucket may be converted from a linked list to a balanced tree for faster lookup inside that bucket—but the mental model stays: array of buckets, each bucket a small structure keyed by hash and disambiguated by equals.

The diagrams below use a 16-bucket row: index 0 is often shown specially because a null key (allowed once in HashMap) is handled as a separate case and conceptually lives in that reserved bucket; all other keys are placed using their hash into some index 0…capacity−1.

70. How LinkedHashMap and LinkedHashSet work internally?

LinkedHashMap (extends HashMap)

LinkedHashMap internally extends HashMap, so it still uses the same hash-table core:

  • Array of buckets — index into the table from the hash
  • Hashing — find which bucket should hold a key
  • Linked list / tree inside a bucket — when several keys collide, they share one bucket; the implementation chains entries (and may use a tree when a chain grows very long, as in modern HashMap)

Additionally, LinkedHashMap keeps a doubly linked list of all entries so iteration can follow insertion order (default) or access order if you construct it that way. That list is separate from “which bucket index,” but every entry node participates in both structures.

Internal structure (fields on each entry)

Each node in LinkedHashMap carries the usual hash-map fields plus order links:

  • int hash
  • K key
  • V value
  • Node<K, V> next — next entry in the same bucket (collision chain)
  • Entry<K, V> before — previous node in the global insertion/access list
  • Entry<K, V> after — next node in that same global list

Extra fields in plain words: before points to the previously inserted (or ordered) entry; after points to the next one in that order. Those pointers maintain insertion or access order while next still handles collisions inside one bucket. In the JDK this is implemented with a concrete entry type that extends HashMap’s node and adds before/after.

LinkedHashSet

LinkedHashSet internally uses a LinkedHashMap. You can think of it as: 

LinkedHashSet  =  LinkedHashMap  +  only keys are used

Internal structure: LinkedHashSet stores each element as a key in that internal LinkedHashMap. The map still needs a value per key, so a dummy object (the same idea as HashSet’s PRESENT) is stored for every key.

Example:

LinkedHashSet<Integer> set = new LinkedHashSet<>();

set.add(10);
set.add(20);
set.add(30);

Internally (conceptually):

10  ->  PRESENT
20  ->  PRESENT
30  ->  PRESENT

A dummy value is stored for every key; the set API only shows you 10, 20, 30.

Indirect question — “array + linked list” vs “array + linked list + doubly linked list”?

If HashMap is “array + linked list,” is LinkedHashMap “array + linked list + doubly linked list”?

Yes, in the usual interview picture — with one important split: the singly linked part (next) is per bucket for collisions; the doubly linked part (before / after) is across the whole map for iteration order. They solve different problems and live on the same entry object.

Structure HashMap — used for what? LinkedHashMap — used for what?
Bucket array Maps hash → bucket index; each slot is the head of (or root of) the structure for keys that landed in that index. Fast random “which drawer?” Same — lookup and put still start here.
Linked list inside a bucket (and tree under heavy collisions) Chains (or trees) keys that collide in the same bucket; next walks that collision structure only. Same — collisions are still handled this way inside each bucket.
Doubly linked list (before / after) — Not present. Iterator order follows bucket index + chains, so order is not insertion order. Links all entries in insertion (or access) order for predictable iteration; does not replace the bucket array or collision next chain.

Short version: array + collision list/tree is for hashing and duplicates inside a bucket; the doubly linked list is only for remembering the order you want when you iterate.

71. How TreeMap works internally?

java.util.TreeMap is a sorted map: keys always come out in order. Inside the JDK it is implemented as a red–black tree — a kind of balanced binary search tree where each node stores a key–value pair and has a colour (red or black) used only to keep the tree balanced after inserts and deletes.

There is no hash table. To put, get, or remove, the map starts at the root and repeatedly asks: is my key smaller or larger than this node’s key? That sends you left or right until you find a match or an empty child slot. Because the tree stays balanced, that path is short — about O(log n) steps.

How ordering is chosen

Natural order (default): if you write new TreeMap<>() with no comparator, keys must implement Comparable (for example Integer sorts 1, 2, 3, …). 

TreeMap<Integer, String> map = new TreeMap<>();

Custom order: pass a Comparator in the constructor if you want another rule (for example reverse order). 

TreeMap<String, String> map = new TreeMap<>(Comparator.reverseOrder());

Each box is one map entry (key + value). Black and Red are only there so the tree can rebalance itself — you still read smaller keys to the left, larger to the right.

Red–black rules (good to know, in plain English)

  • Every node is either red or black.
  • The root is always black.
  • “Leaves” (empty child slots / NIL) count as black.
  • No two reds in a row: a red node’s parent is black.
  • Every path down from a node to those empty leaves passes through the same number of black nodes (black-height rule). That is what keeps the tree from becoming a long skinny line.

Insertion — put(key, value)

New keys are added as a leaf in the right place (smaller left, larger right). New nodes often start red. If that breaks a rule (for example two reds touching), the tree recolours and may rotate a few pointers — you do not do that by hand; TreeMap does it for you.

To find a key, walk from the root and compare at each step — no hashing.

The map finds the node, may replace it with its in-order successor (next key in sorted order) or similar trick, then runs the same kind of recolour + rotate fix-up so red–black rules still hold. You remember: remove → maybe swap with successor → rebalance.

73. Iterator vs ListIterator?

Supported collections

  • Iterator comes from Iterable / Collection. You can use it on List, Set, Queue, and on a Map’s views (keySet(), values(), entrySet()) because those views are collections you iterate.
  • ListIterator exists only for List implementations (for example ArrayList, LinkedList) via list.listIterator().

Direction of traversal

  • Iterator — forward only (hasNext / next).
  • ListIterator — forward and backward (hasNext / next and hasPrevious / previous), plus index helpers on a list.

Real-world analogy

Think of a photo gallery: with a plain Iterator you only have a “Next” button. With ListIterator you get “Next” and “Previous” because a list has a well-defined left-to-right order with positions.

Code — one direction vs both

Iterator (forward only):

List<String> names = new ArrayList<>();
names.add("Dhoni");
names.add("Virat");

Iterator<String> iterator = names.iterator();
while (iterator.hasNext()) {
    System.out.println(iterator.next());
}

ListIterator (forward, then backward):

ListIterator<String> listIterator = names.listIterator();
while (listIterator.hasNext()) {
    System.out.println("Forward: " + listIterator.next());
}
while (listIterator.hasPrevious()) {
    System.out.println("Backward: " + listIterator.previous());
}

Indirect questions

  1. Best way to traverse a Set — why not ListIterator?
    Use Iterator (or an enhanced for-loop, which uses it under the hood). A set has no guaranteed positional order, and ListIterator is tied to List APIs that need indexed / bidirectional list positions.
  2. Safe removal while looping?
    Call Iterator.remove() immediately after next() on the same iterator. Structural changes done any other way during iteration can cause ConcurrentModificationException on fail-fast iterators.
  3. Insert while iterating?
    Use ListIterator.add(E) on a list — it inserts relative to the iterator’s current position (just before the element that would be returned by next()).
  4. Iterate backward through a list?
    Use ListIterator starting at the end (list.listIterator(list.size())) or walk forward first, then use hasPrevious / previous.
  5. What if the collection is modified during iteration the wrong way?
    Fail-fast iterators usually throw ConcurrentModificationException because the iterator detects a structural change it was not told about.
  6. Which types cannot use ListIterator?
    Set, Queue, and Map (directly) — there is no standard list iterator for them because they are not Lists (no list positions / bidirectional list contract).
  7. When prefer Iterator over ListIterator in apps?
    Use Iterator for sets, queues, and map views — it is universal and simple. Use ListIterator when you truly need backward walk, index, or add/set while traversing a list.

74. How do fail-fast iterators work? Why ConcurrentModificationException?

Fail-fast iterators work on the live collection (they read the real structure as you go). The collection keeps a modification count (conceptually: “how many structural changes happened?”).

On every next() (or similar step), the iterator checks whether that count still matches what it expected when it was created. If something else changed the collection structurally (add/remove without using this iterator’s remove()), the iterator fails immediately by throwing ConcurrentModificationException instead of returning stale or inconsistent data.

Goal: catch accidental concurrent or mid-loop structural edits early. Trade-off: you must use the iterator’s own remove() (or avoid structural changes) while iterating.

75. How do fail-safe iterators work? Why no ConcurrentModificationException?

Fail-safe iterators do not walk the backing collection directly. They iterate over a snapshot or copy of the data taken when the iterator was created (or over a weakly consistent view, depending on the class — the interview idea is the same: not the single live structure).

Because of that, changes made to the original collection after the iterator was created are not always visible to the iterator, and it will not throw ConcurrentModificationException for those changes — you are not “cheating” the fail-fast check; you simply are not looking at the same mutable array of entries anymore.

Indirect question — which uses more memory?

Fail-safe approaches usually consume more memory (copy or extra bookkeeping). Fail-fast iterators use little extra memory because they traverse the real collection — but they throw on unexpected structural modification instead of hiding it.

76. Comparable vs Comparator — differences?

Comparable is implemented by your class. It defines the natural ordering of objects of that type (one default “how should instances compare?”).

Comparator is a separate strategy object. You can define many different sort orders without changing the original class — pass different comparators to Collections.sort, List.sort, TreeSet constructors, etc.

Indirect question — both natural order and a custom comparator?

If a class implements Comparable, that is its natural ordering (for example by id or name) when no comparator is supplied.

If you pass a Comparator explicitly (for example Collections.sort(list, comparator) or stream.sorted(comparator)), that comparator wins for that call or that sorted structure. Natural ordering is used only when no Comparator is given (or the API defaults to natural order).

77. List.of() vs Collections.unmodifiableList()?

List.of() (Java 9+)

  • Creates a truly immutable list — no add, remove, or set.
  • Any mutating call throws UnsupportedOperationException.
  • No null elements — passing or having null throws NullPointerException.
  • Often very compact for small fixed lists.
List<String> immutableList = List.of("A", "B", "C");
immutableList.add("D"); // UnsupportedOperationException

Collections.unmodifiableList()

  • Wraps an existing list: the view cannot be modified through the wrapper, but the backing list can still change — and those changes show through the view.
  • Mutating through the unmodifiable view throws UnsupportedOperationException.
  • Null elements follow whatever the backing list allows.
List<String> originalList = new ArrayList<>(List.of("A", "B", "C"));
List<String> unmodifiableList = Collections.unmodifiableList(originalList);
unmodifiableList.add("D"); // UnsupportedOperationException
originalList.add("D");     // OK — backing list changed; view now sees four elements

Rule of thumb: List.of = immutable values you own; unmodifiableList = read-only window onto someone else’s mutable list unless they also stop mutating it.
